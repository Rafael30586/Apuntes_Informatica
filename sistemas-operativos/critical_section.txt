Sección crítica en sincronización

Una sección crítica es una parte de un programa donde recursos compartidos (como memoria, archivos o variables) son accedidas por múltiples procesos o hilos. Para evitar problemas como "race conditions" e inconsistencia de datos, solo un proceso/hilo debería ejecutar la sección crítica a la vez usando técnicas de sincronización. Esto asegura que las operaciones en recursos compartidos son llevadas a cabo de manera segura y predecible.

------------------------
Estructura de una sección crítica

1. Entry section
* El proceso solicita permiso para entrar a la sección crítica
* Herramientas de sincronización (por ejemplo, mutex o semáforo) se usan para controlar el acceso

2. Sección crítica: el verdadero código donde los recursos compartidos son accedidos o modificados

3. Sección de salida: el proceso libera el cierre o semáforo, permitiendo que otros procesos entren en la sección crítica

4. Sección remanente: el resto del programa que no involucra el acceso a recursos compartidos

Problema de sección crítica

* Los recursos compartidos incluyen memoria, variables globales, archivos y bases de datos
* Una "race condition" ocurre cuando 2 o más procesos intentan actualizar datos compartidos al mismo tiempo, llevando  a resultados inesperados. Ejemplo: dos transacciones bancarias modificando el mismo balance de cuenta simultáneamente sin sincronización podría llevar a una balance final incorrecto

Requerimientos de una solución
Una buena solución de sección crítica debe asegurar:
* Correctitud: los datos compartidos deberían permanecer consistentes
* Eficiencia: debería minimizar la espera y maximizar la utilización de CPU
* Justicia (fairness): ningún proceso debería ser injustamente retrasado o "starved"

----------------------------
Requerimientos de soluciones de sección crítica

1. Exclusión mutua
* Como máximo un solo proceso puede estar dentro de la sección crítica a la vez
* Previene conflictos al asegurar que 2 procesos no actualicen el recurso compartido simultáneamente

2. Progreso
* Si ningún proceso està en la sección crítica, y algunos proceso quieren entrar, la elección sobre quien es el próximo en entrar deberìa ser pospuesta indefinidamente
* Asegura que el sistema continúe progresando en lugar de quedarse trabado

3. Espera limitada (bounded waiting)
* Debe haber un lìmite sobre cuanto tiempo espera un proceso antes de que tenga la oportunidad de entrar en la sección crítica
* Previene inanición, donde un proceso es repetidamente sobrepasado mientras otros se ejecutan 

----------------------------
Ejemplos de secciones críticas en aplicaciones del mundo real

* Sistema bancario (ATM o banco online)
  - Sección crítica: actualizando un balance de cuenta durante un depòsito o retiro
  - El problema si no se maneja: dos retiros simultáneos podrían resultar en un balance final incorrecto  debido a "race conditions"
* Sistema de reserva de tickets (aerolíneas, pelìculas, trenes)
  - Sección crítica: reservar el último asiento
  - El problema si no se maneja: a dos usuarios se les podría mostrar el mismo asiento disponible y ambos podrían reservarlo, llevando a sobre-reserva
* Cola de impresión en una impresora conectada en red
  - Sección crítica: enviado de trabajos de impresión a la cola de impresión
  - El problema si no se maneja: los trabajos de impresión podrían mezclarse o saltearse si múltiples usuarios envían trabajos simultáneamente
* Edición de archivos en documentos compartidos (por ejemplo: Google docs, MS Word con acceso compartido)
  - Sección crítica: guardar o escribir en el documento compartido
  - El problema si no se maneja: ediciones simultàneas podrían conflicto en versiones o pèrdida de datos
