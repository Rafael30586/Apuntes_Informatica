El rango de direcciones virtuales que el sistema operativo asigna a un usuario o programa en ejecución se llama address space. Esta es el área de direcciones virtuales contiguas disponibles para ejecutar instrucciones y almacenar datos.
El rango de direcciones virtuales en un address space comienza con cero y se puede extender a la dirección más alta permitida por la arquitectura del sistema operativo.
El sistema operativo provee a cada usuario con un único address space y mantiene la distinción entre los programas y datos pertenecientes a cada address space. Dentro de cada address space, el usuario puede comenzar múltiples tareas, usando bloques de control de tareas o TCBs que permiten la multiprogramación. 
En cierto sentido los address space de un sistema operativo es como un proceso de UNIX, y el identificador de address space (ASID) como un id de proceso (PID). Además, los TCBs son como hilos de UNIX en el que cada sistema operativo soporta el procesamiento de múltiples instancias de trabajo de manera concurrente.
De todas maneras, el uso de múltiples address spaces virtuales en sistemas operativos tiene ciertas ventajas. El virtual addressing permite un rango de direcciones que es mayor que las capacidades de almacenamiento centrales del sistema. El uso de múltiples address spaces virtuales provee esta capacidad de virtual addressing a cada trabajo en el sistema al asignar a cada trabajo su propio address space virtual. El potencialmente enorme número de address spaces provee al sistema con una gran capacidad de virtual addressing.
Con múltiples address spaces virtuales, los errores se pueden limitar a un solo address space, excepto por errores "in commonly addressable storage", en consecuencia se mejora la fiabilidad del sistema y se hace más fácil la recuperación de errores. Los programas en diferentes address spaces están protegidos el uno del otro. Aislar los datos en su propio address space también protege los datos.
Los sistemas operativos usan muchos address spaces. Hay por lo menos un address space para cada trabajo en progreso y una para cada usuario ingresado.
El uso de address spaces permite al sistema operativo mantener una distinción entre los programas y datos a cada address space. Las áreas privadas en el addrsss space de un usuario están aisladas de las áreas privadas en otros address spaces, y este aislamiento provve gran parte de la seguridad (security) en el sistema operativo.
Aún así, cada address space también contiene un área común que es accesible a todos los otros address spaces. Como mapea todas las direcciones disponibles, un address space incluye código y datos de sistema así como código y datos de usuario. En consecuencia, no todas las direcciones mapeadas están disponibles para el código y datos de usuario.
La habilidad de que muchos usuarios compartan los mismos recursos conlleva la necesidad de proteger a los usuarios los unos de los otros y proteger al propio sistema operativo. Junto con métodos como "claves" para proteger almacenamiento central y palabras de código para proteger archivos y programas, direcciones separadas aseguran que los programas y datos de usuario no se superpongan.
Las aplicaciones necesitan que haya comunicación entre address spaces. Los sistemas operativos proveen dos formas de comunicación:
* Programando (scheduling) un "service request block" (SRB), un proceso asincrónico
* Usando servicios "cross memory" y "access registers", un proceso síncrono
Un programa usa un SRB para iniciar un proceso en otro address space o en el mismo address space. EL SRB es asíncrono por naturaleza y se ejecuta independientemente del programa que lo emite, así mejorando la disponibilidad de recursos en un entorno de multiprocesamiento.
Un programa usa servicios "cross memory" para acceder al address space de otro usuario directamente.
