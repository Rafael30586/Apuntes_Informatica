Comunicación entre procesos (IPC)

IPC es un mecanismo que permite a los procesos comunicarse y compartir datos entre sí mientras se están ejecutando. Como cada proceso tiene su propio espacio de memoria, IPC provee métodos controlados para intercambiar información y coordinar acciones. Ayuda a que los procesos trabajen juntos eficientemente y de forma segura en un sistema operativo.
* Ayuda a los procesos a sincronizar sus actividades, compartir información y evitar conflictos mientras se comparten recursos
* Hay dos métodos de IPC, "shared memory" y "message passing". Un sistema operativo puede implementar ambos métodos de comunicación

Ejemplo: un sistema ATM de banco, donde un proceso lee la tarjeta y el PIN, otro revisa el balance de cuenta y un tercero dispensa plata. Estos procesos se comunican y coordinan para completar la transacción correctamente.

-------------------------
Shared memory

La comunicación entre procesos usando memoria compartida requiere que los procesos compartan alguna variable y depende completamente de como el programador los implementará. Los procesos pueden usar memoria compartida para extraer información como un registro desde otro proceso y para enviar cualquier información específica a otros procesos.

https://media.geeksforgeeks.org/wp-content/uploads/20250829161702611682/Shared_Memory.webp

* En la imagen de arriba, un espacio común de memoria es asignado por el kernel
* El proceso A escribe datos en la región de la memoria compartida (paso 1)
* El proceso B puede directamente leer estos datos desde la misma región de memoria compartida (paso 2)
* Ya que ambos procesos acceden al mismo segmento de memoria, este método es rápido pero requiere mecanismo de sincronización (como semáforos) para evitar conflictos cuando múltiples procesos escriben/leen simultáneamente
* Ejemplo: múltiples personas pueden editar un documento al mismo tiempo en Google doc 

-------------------------
Message passing

Message passing es un método en el cual los procesos se comunican al enviar y recibir mensajes para intercambiar datos. Un proceso envía un mensaje y el otro lo recibe, permitiendo que ambos compartan información. El message passing puede realizarse a través de diferentes métodos como sockets, colas de mensajes o pipes.

https://media.geeksforgeeks.org/wp-content/uploads/20250829161702509451/Message_Passing_.webp

* En la imagen de arriba, los procesos intercambian información al enviar y recibir mensajes a través del kernel
* El proceso A envía un mensaje al kernel (paso 1)
* El kernel luego entrega el mensaje al proceso B (paso 2)
* Acá, los procesos no comparten memoria directamente. En su lugar, la comunicación ocurre vía system calls (send(), recv() o similar)
* Este método es más simple y seguro (safer) que shared memory porque no hay riesgo sobreescribir datos, pero hay más sobrecarga debido a la intervención del kernel
* Ejemplo: múltiples personas envían actualizaciones a un grupo de chat, pero cada mensaje va a través del servidor antes de que otros lo vean, como procesos enviando mensajes en lugar de compartir directamente memoria

Problemas en IPC

IPCs enfrentan desafíos cuando múltiples procesos comparten recursos. Una sincronización inapropiada puede causar "race coditions", interbloqueo (deadlock) e inanición (starvation), y los datos compartidos podrían sufrir inconsistencia de datos. IPC también añade sobrecarga y aumenta los problemas de seguridad. Gestionar muchos procesos puede llevar a problemas de escalabilidad

Algunos problemas comunes y clásicos de IPC son:
* Dining philosophers problem: este problema ilustra deadlock e inanición (stravation). Involucra a 5 filósofos sentados alrededor de una mesa, cada uno necesita dos tenedores (recursos compartidos) para comer. Si todos los filósofos agarran un tenedor al mismo tiempo, ninguno puede comer, resultando en deadlock
Solución: 
- usar semáforos o monitores para controlar el acceso a los tenedores
- permitir que solo un filósofo agarre tenedores a la vez o limitar la comida a 4 filósofos
- hacer cumplir una orden de elegir tenedores para evitar espera circular
- previene deadlock y onanición

* Producer-consumer problem: este problema trata con sincronización y gestión del buffer. Describe a productores generando datos y colocándolos en un buffer compartido, mientras unos consumidores remueven datos de él. El desafío principal es prevenir que los productores añadan datos a un buffer lleno y que los consumidores remuevan datos de una buffer vacío
Solución: 
 - usar mutex para asegurar la mutua exclusión en el buffer compartido
 - usar "counting semaphores" para rastrear ranuras de buffer vacías y llenas
 - el productor espera si el buffer está lleno; el consumidor espera si el buffer está vacío
 - asegura sincronización apropiada y consistencia de datos

* Readers-writers problem: este problema se enfoca en acceso concurrente para datos compartidos. Permite que múltiples lectores lean datos simultáneamente, mientras los escritores requieren acceso exclusivo. El desafío está en evitar la inanición tanto en lectores como en escritores
Solución:
 - usar trabas lector-escritor o semaphores
 - permitir que múltiples lectores lean simultáneamente
 - otorgar a los escritores acceso exclusivo o datos compartidos
 - aplicar reglas de prioridad para evitar inanición

* Sleeping Barber problem: este problema demuestra coordinación de procesos. Presenta a un peluquero que duerme cuando no hay clientes y es despertado cuando un cliente llega y una silla está disponible. La dificultad radica en gestionar sillas y clientes correctamente
Solución:
  - usar semaphores para gestionar llegada de clientes y disponibilidad del peluquero
  - el peluquero duerme cuando no hay clientes presentes
  - los clientes esperan si hay sillas disponibles, sino se van
  - asegura una coordinación de procesos apropiada sin deadlock
