Niveles de aislamiento en bases de datos explicados

Cuando múltiples usuarios interactúan con una base de datos al mismo tiempo, mantener la consistencia de datos se vuelve un desafío serio. Imaginá miles de transacciones actualizando balances, colocando órdenes o modificando registros simultáneamente. Sin el aislamiento apropiado, los datos pueden volverse fácilmente inconsistentes, corruptos o lógicamente incorrectos
Acá es en donde los niveles de asilamiento de transacciones entran en juego. El asilamiento define como y cuando los cambios hechos por un transacción se vuelven visibles para otros. Afecta directamente el comportamiento, rendimiento y correctitud de datos del sistema. Entender los niveles de aislamiento es esencial para desarrolladores backend, ingenieros de bases de datos y arquitectos de sistema que construyan sistemas fiables y escalables

--------------------------------------
¿Qué es aislamiento de transacciones?

Una transacción es una secuencia de operaciones ejecutadas como una única unidad de trabajo. En bases de datos relacionales, las transacciones siguen los principios ACID: atomicidad, consistencia, aislamiento y durabilidad
El aislamiento determina que tan independientes son las transacciones entre sí mientras corren de manera concurrente. En sistemas reales, un aislamiento completo puede reducir rendimiento, mientras que el bajo aislamiento puede introducir errores lógicos. Por lo tanto, las bases de datos proveen diferentes niveles de aislamiento que equilibran correctitud y rendimiento
Antes de explorar cada nivel, es importante entender los principales problemas de concurrencia que están diseñados para prevenir

-------------------------------
Problemas comunes de concurrencia

Una lectura sucia o dirty read ocurre cuando una transacción lee datos que han sido modificados por otra transacción pero todavía no confirmados. Si la segunda transacción hace roll back, la primera transacción ha leído datos inválidos
Una non-repeatable read ocurre cuando una transacción lee la misma fila dos veces y obtiene diferentes valores porque otra transacción actualizó y confirmó cambios entre las dos lecturas
Una lectura fantasma aparece cuando una transacción consulta un conjunto de filas coincidiendo con una condición, y otra transacción inserta o borra filas que coinciden con esa condición antes de que la primera transacción finalice
Estas anomalías definen las diferencias entre los diferentes niveles de aislamiento

-------------------------------
Read uncommitted

Read uncommitted es el nivel de aislamiento más bajo. Las transacciones pueden ver los cambios hechos por otras transacciones incluso si esos cambios no están confirmados todavía
Por ejemplo, la transacción A actualiza el crédito de un usuario de 100 a 50 pero todavía no lo ha confirmado. La transacción B lee el crédito y ve 50. Si la transacción A hace rollback, el crédito retorna a 100, pero la transacción B ya uso datos incorrectos. Esta es una clásica lectura sucia
Este nivel ofrece máximo rendimiento porque prácticamente no hay restricciones de cierre, pero sacrifica correctitud de datos. En la práctica, la mayoría de las bases de datos modernas no permiten lecturas sucias incluso si este nivel está seleccionado

----------------------------
Read committed

Read commited previene de lecturas sucias. Una transacción solo puede ver datos que han sido confirmados por otras transacciones 
Por ejemplo, la transacción A actualiza el precio de un producto pero no confirma. La transacción B intenta leer ese producto. En lugar de ver el valor sin confirmar, espera o lee la última versión confirmada
De todas formas,  non-repeateble reads pueden aún ocurrir. Suponga que la transacción B lee el crédito de un usuario como 100. Mientras tanto, la transacción A lo actualiza a 120 y confirma. Si la transacción B lee el crédito otra vez, ahora lee 120. La misma consulta produjo resultados diferentes dentro de la misma transacción
Read committed es el nivel de aislamiento por defecto en muchas bases de datos, incluyendo PostgreSQL y Oracle. Provee un buen equilibrio entre consistencia y rendimiento

------------------------
Repeatable read

Repeatable read asegura que si una transacción lee una fila, será siempre la misma versión de esa fila por la duración de la transacción
Por ejemplo, la transacción A lee una orden con una cantidad de 200. La transacción B actualiza esa orden a 250 y confirma. Si la transacción lee la misma orden otra vez, todavía ve 200 hasta que completa. Esto elimina los non-repeatable reads
De todas formas, las lecturas fantasma puede ocurrir dependiendo de la implementación de la base de datos. Por ejemplo, la transacción A selecciona todas las órdenes con el estado "pendiente". La transacción B inserta una nueva orden pendiente y confirma. Si la transacción A corre la misma consulta otra vez, podría llegar a ver la nueva fila
En el motor InnoDB de MySQL, repeatable read también previenen lecturas fantasma usando gap locking. En PostgreSQL, la prevención fantasma es manejada de forma diferente usando mecanismos MVCC
Repeatable read provee una consistencia más fuerte pero podría incrementar la sobrecarga de cierre (locking) y reducir concurrencia 

-------------------------
Serializable

Serializable es el nivel de aislamiento más alto. Garantiza comportamiento equivalente a transacciones corriendo una después de la otra de manera secuencial, incluso si se ejecutan de manera concurrente. 
Por ejemplo, dos usuarios tratan de transferir dinero a la misma cuenta al mismo tiempo. Bajo niveles de aislamiento más bajos, ambas transacciones podrían leer el mismo crédito y salirse del presupuesto. Bajo el nivel de aislamiento serializable, la base de datos asegura que una transacción se completa primero, y la segunda opera en datos actualizados o falla y reintenta
Este nivel previene de lecturas sucias, non-repeatable reads y lecturas fantasma. De todas formas, puede reducir de manera significativa el throughput porque requiere cierre estricto o mecanismos de control de concurrencia avanzada
Srializable se usa generalmente en sistemas financieros o lógica de negocio crítica donde la correctitud absoluta es más importante que el desempeño

--------------------------
¿Cómo hacen las bases de datos para implementar aislamiento?

Las bases de datos modernas usan generalmente mecanismos de cerrado o MVCC (multi-version concurrency control) 
El cerrado bloquea otras transacciones de modificar o leer ciertos datos. Aunque es conceptualmente simple, un cerrado excesivo puede reducir el desempeño
MVCC trabaja de manera diferente. En lugar de bloquear lectores, mantiene múltiples versiones de las filas. Cada transacción ve una foto consistente de la base de datos. PostgreSQL se fía fuertemente en MVCC, mientras que MySQL InnoDB combina MVCC con cerrado
Entender como tu base de datos específica implementa aislamiento es crucial porque el comportamiento puede ser un poco diferente a través de los sistemas
