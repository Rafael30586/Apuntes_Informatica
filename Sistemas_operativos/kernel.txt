El kernel es el núcleo del sistema operativo. Funciona como puente entre las aplicaciones de software y el hardware de una computadora.
* El kernel gestiona recursos del sistema, como el CPU, la memoria y dispositivos, asegurando de que todo trabaje en conjunto de manera fluida y eficiente.
* Gestiona tareas como programas en ejecución, acceso a archivos y conexión a dispositivos como impresoras y teclados.

https://media.geeksforgeeks.org/wp-content/uploads/20250124124411692602/kernel.webp

* Asegura multitareas eficientes y seguras.
* Gestiona la estabilidad del sistema y previene de acceso a recursos no autorizados.

Tipos de kernel
* Monolítico: todos los servicios del sistema operativo se ejecutan en espacio de kernel. Rápidos pero menor aislamiento de fallas. Ejemplos: UNIX, Linux, etc.
* Microkernel: funcionalidad de kernel mínima. La mayoría de los servicios se mueven a espacio de usuario. Mejor confiabilidad. Ejemplos: Minix 3 y Mach.
* Kernel híbrido: Mezcla ideas de monolítico y microkernel. Algunos servicios en kernel por rapidez y otros aislados por seguridad. Ejemplos Familia NT de Windows (2000, XP, Vista, 7, 8, 10, etc), macOS.
* Nanokernel: un kernel extremadamente pequeńo, proveyendo solamente abstracción básica de hardware, todo lo demás afuera.
* Exokernel: separa protección vs gestión; les da a las aplicaciones control directo sobre abstracciones de hardware.

Funciones del kernel

https://media.geeksforgeeks.org/wp-content/uploads/20250725113119973417/Function-Of-Kernel.webp

1. Gestión de procesos: programación y ejecución de procesos.
2. Gestión de memoria: asignación y liberación de espacio de memoria, gestión de memoria virtual y manejo de protección de memoria. 
3. Gestión de dispositivos: Gestión de dispositivos de entrada y salida, proveyendo una interfaz unificada para dispositivos y manejando comunicación con controladores de dispositivos.
4. Gestión de archivos de sistema: Gestionando operaciones de archivos y proveyendo una interfaz de sistema de archivos a aplicaciones.
5. Gestión de recursos: Gestionando recursos de sistema (CPU, espacio en disco, ancho de banda de red). Principalmente asignando y liberando recursos según la necesidad.
6. Control de acceso y seguridad: haciendo cumplir políticas de control de acceso como permisos de usuario y autenticación.
7. Comunicación interprocesos: Facilitando comunicación entre procesos proveyendo mecanismos como mensajes y memoria compartida. 
  
Modo de trabajo del kernel
* El kernel es la primera parte del sistema operativo cargada en la memoria durante el inicio, y se mantiene residente mientras el sistema está en ejecución.
* Opera en un modo privilegiado (modo kernel), separado del modo usuario para aplicaciones; las aplicaciones de usuario no pueden acceder directamente al hardware o recursos críticos.
* Las aplicaciones realizan solicitudes al kernel vía "system calls" (o interrupciones de software). El kernel maneja estos cambiando entre modo usuario y modo kernel.
* El kernel ejecuta la operación solicitada, por ejemplo: I/O de archivos, creación de procesos, asignación de memoria, etc.
* Al terminar, el kernel retorna el resultado (o error) al espacio de usuario.

Kernel space y user space
Son regiones segregadas de memoria virtualizada designadas para mejorar la seguridad y estabilidad del sistema operativo. El kernel space tiene acceso total al hardware ejecutándose en modo privilegiado, mientras que el user space ejecuta aplicaciones en modo sin privilegios, previniendo que el sistema se rompa por apps que funcionan mal.

https://blogs.oracle.com/linux/userspace-vs-kernelspace-understanding-the-divide

Userspace
También llamada userland, es simplemente "el espacio controlado por el usuario". La mayor parte de las funcionalidades del sistema están implementadas en el userspace, incluyendo: ejecución de aplicaciones (navegadores web, software de oficina y juegos); soporte para frameworks y bibliotecas (gráficos, audio y bibliotecas de redes); servicios de sistema (sistemas init, logging en el sistema y gestión de dispositivos).
Los programas del userspace se ejecutan en modo usuario. En Linux, el kernel provee userpace con un entorno sandbox, llamado modo usuario, en el cual los usuarios pueden actuar libremente. Cada proceso del userspace obtiene su propia región de memoria independiente, llamado espacio de dirección o addres space. Para acceder a cualquier área de memoria, el userspace debe ir a través del kernel. Si el kernel se niega, los usuarios no pueden hacer nada. Por defecto cada proceso solo puede ver la memoria en su propio address space, "no saben que otros existen".

Kernelspace
Es el dominio del propio kernel. El kernel puede ser pensado como el puente entre el hardware y el software. En linux, el kernel tiene su propio address space, independiente de todos los procesos de los usuarios, y puede ver toda la otra memoria en el sistema. A esto se le llama modo supervisor.
En el modo supervisor el kernel lleva a cabo tareas críticas como: gestión de procesos, crea, programa y termina procesos; gestión de memoria, asigna y libera memoria para aplicaciones; gestión de dispositivos, interactúa con discos, placas de red, etc; reforzamiento de seguridad: refuerza control de acceso, autenticación y autorización.
Aunque el kernel tiene acceso no restringido a recursos de sistema, no hace mucho menos que reciba instrucciones. Para el usuario promedio, el kernel no es demasiado útil por sí mismo: su función principal es que las aplicaciones en userspace se ejecuten en un entorno seguro y estable.

La división: interacción userspace-kernelspace
El límite entre userspace y kernelspace es un aspecto crítico de diseńo de un sistema operativo. Para asegurar la estabilidad y seguridad del sistema, el kernel debe controlar cuidadosamente las interacciones entre aplicaciones del userspace y el kernel en sí mismo.
Por el lado del software, para facilitar la interacción del userspace con el kernel, Linux provee un conjunto de userspace APIs (UAPIs). Cuando una aplicación de userspace necesita acceder a un servicio proveído por el kernel, como leer desde un archivo o enviar un paquete de red, debe usar una de esas UAPIs: un system call. Cada vez que un system call es gatillado, Linux cambia de ejecutar código en modo usuario a ejecutar código en modo supervisor.
Por el lado del hardware, para reforzar la división, las arquitecturas de CPU implementan un sistema de anillos de protección. Anillo 0, sinónimo de modo supervisor, permite al código ejecutar cualquier instrucción. Anillo 3, sinónimo de modo usuario, tiene muchas limitaciones para prevenir que los usuarios causen dańo al sistema.

Beneficios de la división
La separación entre userspace y kernelspace tiene efectos significativos en el diseńo de sistema, desempeńo y seguridad. Seguridad: La división previene a las aplicaciones del userspace el acceso a código o datos sensibles del kernel. Reduciendo el riesgo de compromisos del sistema. Estabilidad: La separación limita el acceso de las aplicaciones del userspace a los recursos del kernel, haciendo que el sistema sea más resiliente a "crashes" o mal comportamiento. Desempeńo: El kernel puede optimizar los system calls e interfaces para mejorar el rendimiento del sistema.

Kernel mode y user mode
Kernel mode y user mode son dos estados clave en un sistema operativo. El modo usuario ejecuta aplicaciones regulares con acceso limitado a recursos del sistema, mientras que el modo kernel ejecuta en núcleo del sistema operativo con control total sobre el hardware. Esta separación asegura seguridad, estabilidad y eficiente gestión de recursos. 

https://media.geeksforgeeks.org/wp-content/uploads/20250807124621795398/user_mode.webp

Modo usuario
Es un entorno restringido donde las aplicaciones corren. Cuando un programa comienza, el sistema operativo crea un proceso separado y le asigna su propio espacio de memoria.
Los programas en modo usuario no pueden acceder al hardware o la memoria del kernel directamente. Deben solicitar acceso a través de system calls al kernel.

Características clave del modo usuario
* Entorno aislado
* Acceso al harware solo a través del kernel
* Ejecución segura: Errores o "crasheos" no afectan al sistem entero

Ventajas
* Más fácil de debuggear: los bugs son contenidos en aplicaciones individuales
* "Crash containment": Solo la aplicación rota es afectada
* Aislamiento de procesos: Una aplicación no puede corromper la memoria de otra
* Acceso controlado: El kernel revisa y autoriza solicitudes
* Estabilidad y confiabilidad: las aplicaciones no pueden afectar directamente al kernel u otros procesos, reduciendo el riesgo de rupturas del sistema completo
* Securidad: el acceso directo a recursos críticos está bloqueado, previniendo uso no autorizado.
