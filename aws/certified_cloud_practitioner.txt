¿Qué es cloud computing? Es la práctica de usar una red de servidores remotos hospedados en internet para almacenar, gestionar y procesar datos, en lugar de hacerlo en una computadora personal o un servidor local.

Evolución del hospedaje en la nube:
- Servidor dedicado: una máquina física dedicada a un solo negocio. Corre una sola aplicación web. Es muy caro, requiere mucho mantenimiento y  es de alta seguridad.
- Servidor privado virtual o VPS: Una máquina física dedicada a un solo negocio. La máquina física es virtualizada en sub máquinas. Corre múltiples aplicaciones web. Mejor utilización y aislamiento de recursos.
- Hospedaje compartido o shared hosting: Una máquina física compartida por cientos de negocios. Muy barato, funcionalidad limitada y pobre aislamiento.
- Hospedaje en la nube o cloud hosting: múltiples máquinas físicas que actúan como un solo sistema. El sistema es abstraído en múltiples servicios en la nube. Es flexible, escalable, seguro, altamente configurable y "cost-effective".

Amazon: es una corporación multinacional americana con su sede en Seattle. Fue fundad en 1994 por Jeff Bezos y comenzó como un atienda en línea de libros y se expandió a otros productos.

AWS: son los servicios de la nube de amazon. Fue lanzado en 2006.

Simple Queue Service o SQS: fue el primer servicio de AWS lanzado al público.

Simple Storage Service o S3: fue lanzado en marzo de 2006.

Elastic Compute Cloud o EC2: lanzado en agosto de 2006.

En 2010 se reportó que los sitios de web de "retail" de amazon.com migraron a AWS.

En abril de 2003 Amazon empezó a ofrecer programas de certificación para ingenieros en computación.

Proveedores de servicios en la nube (CSPs):
- proveen múltiples servicios en la nube, decenas a centenas de servicios;
- esos servicios en la nube pueden ser encadenados entre sí para crear arquitecturas en la nube;
- esos servicios en la nube son accesibles vía una sola API unificada, por ejemplo, AWS API.
- esos servicios en la nube tienen  una oferta  de infraestructura como un servicio (IaaS).
- esos servicios en la nube utilizan facturación medida basada en uso (por hora, por segundo, etc).
- esos servicios tienen rico monitoreo integrado, ejemplo AWSCLoudTrail
- esos servicios ofrecen automatización vía infraestructura como código (IaC)
Si una compañía ofrece múltiples servicios en la nube bajo un a misma interfaz de usuario pero no cumple con la mayoría de estos requisitos, estaríamos hablando de una plataforma en la nube o cloud platform, ejemplos: Twilio, HashiCorp, Databricks

CPSs:
- Top tier: AWS, Microsoft Azure, Google Cloud Platform, Alibaba Cloud.
- Mid tier: IBM Cloud, Oracle Cloud, Huawei Cloud, Tencent Cloud.
- Light Tier: Vultr, Digital ocean, Akamai Connected Cloud (Linode).
- Private tier: OpenStack (Rackspace), Apache CloudStack, *Vmware vSphere.

Un proveedor de servicios en la nube puede tener cientos de servicio en la nube que están agrupados en varios tipos de servicios. Los cuatro tipos de servicios cloud más comunes para infraestructura como un servicio (IaaS) serían:
- computar: una computadora virtual en la que puede correr una aplicación, programas y código.
- redes: podría ser una red virtual que defina concexiones de internet o aislamientos de red entre servicios o saliente al internet.
- almacenamiento: podría ser un disco duro virtual que almacena archivos.
- bases de datos: podría ser una base de datos virtual pata guardar datos de reportes o una base de datos para una aplicación web de propósito general.

Evolución de la computación en la nube:
- servidor dedicado:
* un servidor físico utilizado por un cliente.
* tenés que adivinar la capacidad
* se paga de más porque hay mucho que no se utiliza.
* no se puede escalar verticalmente, hace falta una migración manual.
* reemplazar un servidor es muy difícil.
* estás limitado por el sistema operativo huésped.
* múltiples aplicaciones pueden resultar en conflictos al compartir recursos.
* tenés garantía de seguridad, privacidad y completa utilización de recursos subyacentes.
- máquinas virtuales:
* se puede ejecutar varias máquinas virtuales en una sola máquina física.
* el hypervisor es el software que permite ejecutar las MVs.
* un servidor físico compartido por varios clientes.
* se paga por una fracción del servidor.
* se paga de más por una máquina virtual porque hay espacio que está sin usar.
* estanos limitados por el sistema operativo huésped.
* varias aplicaciones en una misma máquina virtual pueden resultar en conflictos al compartir recursos.
* es fácil exportar o importar imágenes para la migración.
* es fácil de escalar vertical u horizontalmente.
- contenedores:
* máquinas virtuales ejecutando varios contenedores.
* Docker Daemon es el nombre de la capa de software que permite ejecutar múltiples contenedores.
* se puede maximizar la utilización de la capacidad disponible lo cual es mejor para el teme del costo.
* los contenedores comparten el mismo sistema operativo subyacente por lo que los contenedores son más eficientes que múltiples máquinas virtuales.
- funciones:
* son MVs gestionadas ejecutando contenedores gestionados.
* conocido como computación serverless
* subís una pieza de código, elegís la cantidad de memoria y duración.
* solo responsable por el código y los datos.
* muy buen costo, solo pagás por el tiempo en el que el código está en ejecución, las MVs solo están en funcionamiento cuando hay código por ejecutar.

Tipos de computación en la nube:
* SaaS o software as a service: es un producto gestionado por el proveedor de servicio. No nos preocupamos de como el servicio es mantenido. Funciona y se mantiene disponible. Ejemplos: Salesforce, Gmail, office 365. Diseñado para clientes.
* PaaS o platform as a service: se enfoca en el despliegue y gestión de nuestras aplicaciones. No hay que preocuparse por configurar o entender el hardware o el S.O. Ejemplos: Heroku, Elastic beanstalk. Pensados para desarrolladores.
* IaaS o infraestructure as a service: provee acceso a características de redes, computadoras y espacio para almacenar datos. No hay que preocuparse del personal IT, centros de datos y hardware. Ejemplos: Microsoft azure, AWS y Oracle cloud. Pensados para administradores.

Modelos de despliegue en computación en la nube:
* Public cloud: todo está construido sobre el proveedor de servicios en la nube. También conocido como cloud native o cloud first.
* Private cloud: todo está construido sobre datacenters de la compañía. También conocido como on-premise. La nube podría ser OpenStack.
* Híbrido: usa los dos anteriores. Hay una conexión (podría ser VPN) entre el datacenter y el VPC del CSP.
* Cross-cloud: usa múltiples proveedores de la nube.

Casos de uso de los modelos de despliegue en la nube:
* Public cloud: compañías que están empezando ya o que son lo bastante pequeñas como para hacer el paso de VPS a CSP. Startups, ofrecimiento de SaaS,  nuevos proyectos y compañías. Ejemplo: Dropbox.
* Híbrido: Organizaciones que empezaron con su propio datacenter. No pueden moverse completamente a cloud debido a esfuerzo de migración o cumplimiento de seguridad. Bancos, Fintech, grandes proveedores de servicios profesionales.
* On-Premise: organizaciones que no pueden funcionar en cloud debido a cumplimiento de regulación estricto o por el tamaño de su organización. Sector público (gobierno), hospitales con información muy sensible, empresas de seguros.

Beneficios para una empresa que migre a cloud:
* Incrementar velocidad y agilidad.
* Pay-as-you-go pricing: trade capital expense for variable expense.
* Economy of scale: Benefit from massive economies of scale.
* Alcance global en minutos.
* Seguridad.
* Confiabilidad: no hace falta gastar dinero en correr y mantener datacenters.
* Alta disponibilidad.
* Escalabilidad.
* Elasticidad.

Seis ventajas del cloud:
* Cambiá "capital expense" por "variable expense": Pagás on-demand lo que significa que pagás por lo que consumís por horas, por minutos o por segundos.
* Beneficios de economías masivas de escalado: compartís el costo con otros clientes para ahorrar mucho. Cientos de miles de usuarios usando una fracción de un servidor.
* No hace falta adivinar la capacidad: lanzar o destruir servicios cuando queramos en lugar de pagar por servidores sub utilizados.
* Incrementar velocidad y agilidad: lanzar recursos en unos poco minutos con unos pocos clicks.
* No hace falta gastar dinero en ejecutar y mantener datacenters: enfócate en tus propios clientes, desarrollando y configurando tus aplicaciones.
* Volvete global en minutos: desplegá tu aplicación en múltiples regiones alrededor del mundo con unos pocos clicks. Se provee latencia más baja y una mejor experiencia para los clientes a un costo mínimo.

Infraestructura Global De AWS
------------------------------------------------------------------------------------------------------

¿Qué es la infraestructura global de AWS? es hardware y datacenters que están globalmente distribuidos y conectados entre sí para funcionar como un gran recurso para los clientes.
La infraestructura global de AWS está hecha de los siguientes recursos:
* 32 launched regions
* 102 zonas de disponibilidad
* 115 lugares de conexión directa
* más de 550 puntos de presencia
* 35 zonas locales o Lcal zones
* 29 wavelength zones
Ejemplo: en la región de US oeste (Oregon) hay 4 zonas de disponibilidad.

Regiones: son ubicaciones que consisten en una o más zonas de disponibilidad.
Cada región está físicamente aislada de las otras regiones en términos de ubicación, energía y suministro de agua.
La primera región fue US-East 1 (Northern Virginia). Casi todos los servicios se hacen disponibles primero en esta región.
El costo de los servicios varían según la región.
Al elegir una región hay cuatro factores a considerar:
* Las regulaciones de la región.
* El costo de los servicios de AWS en la región.
* Qué servicios de AWS están disponibles.
* Cual es la latencia o distancia con respecto a los usuarios.
No todos los servicios de AWS están disponibles en todas las regiones.

Los servicios de AWS pueden ser globales o regionales.
Los servicios globales operan a través de múltiples regiones y están fijados como globales. Ejemplos: Amazon S3, CloudFront, Route53, IAM.

Zonas de disponibilidad (availability zones): son ubicaciones físicas para uno o más data centers.
Un datacenter es un edificio que contiene cientas o miles de computadores.
Los datacenters de una región están aislados entre sí (están en diferentes edificios), pero estarán lo suficientemente cerca como para que haya poca latencia (<10ms).
Las zonas de disponibilidad están representadas por un código de región, seguido por una letra, ejemplo: us-east-1a.
Una subred está asociada a una zona de disponibilidad.
Nunca elegimos una zona de disponibilidad al lanzar recursos, lo que hacemos es elegir una subred que está asociada a una AZ (zona de disponibilidad).
Todas las AZs de una región están interconectadas por una fibra dedicada que provee alto rendimiento, con alto ancho de banda, baja latencia, etc.
Todo el tráfico entre las AZs está encriptado.
Las AZs están alejadas unos 100km la una de la otra.

Dominio de fallas o fault domain: es una sección de una red que es vulnerable a daños si un dispositivo crítico o sistema falla. Su propósito es que si un fallo ocurre no irá en cascada fuera del dominio, limitando el daño posible.
Se puede tener dominios de fallas anidados en otros dominios de fallas.
Un nivel de fallas o fault level es una colección de dominios de fallas.
Una región AWS sería un nivel de fallas y un zona de disponibilidad sería un dominio de fallas.
Cada región de Amazon está completamente aislada de otras regiones de Amazon. Esto produce la máxima estabilidad y tolerancia a fallos posible.
Cada AZ está aislada, pero las AZs de una región están conectadas entre sí mediante enlaces de baja latencia.
Cada AZ está diseñada como una zona de fallos independiente.

Red global de AWS: representa la interconexión entre la infraestructura global de AWS. Comúnmente conocido como la columna vertebral de AWS. Puede pensárselo como una autopista, donde las cosas pueden moverse rápidamente entre datacenters.
VPC endpoints: aseguran que los recursos permanezcan en la red de AWS y no lleguen hasta la internet pública.
Edge locations: pueden funcionar como rampas o como vias de escape a la red global de AWS.
AWS Global Accelerator y AWS S3 Transfer Acceleration: Usa Edge locations para alcanzar recursos de AWS en otras regiones atravesando rápidamente la red global de AWS.
Amazon CloudFront (CDN): Usa Edge locations como una via de escape para proveer al Edge storage y computar cerca del usuario final.

Edge locations: son integrales a la infraestructura de AWS permitiendo el envío de contenido y aplicaciones con baja latencia y alto rendimiento. Funcionando como datacenters al borde de la red estas ubicaciones cachean contenido, procesan datos y traen los servicios cloud más cerca de los usuarios finales. Son datacenters estratégicamente posicionados alrededor del mundo diseñados para optimizar el envío de contenido y aplicaciones. Funcionan como puntos de presencia para los servicios de AWS permitiendo que los datos sean servidos desde la ubicación más cercana al usuario final lo cual reduce significativamente la latencia.

https://www.geeksforgeeks.org/devops/aws-edge-locations/

Puntos de presencia o PoP (points of presence): es una ubicación intermedia entre una región de AWS y el usuario final. Esta ubicación podría ser un datacenter o una colección de hardware.
Para AWS un PoP es un datacenter poseído por AWS o un socio de confianza que es utilizado por servicios de AWS relacionados por envío de contenido o subida acelerada (expediated upload).
Edge locations: son datacenters que guardan cache de los archivos más populares (páginas web, imágenes y videos) para que el envío a distancia a los usuarios finales se reduzca.
Regional Edge locations: son datacenters que guardan caches mucho más grandes de archivos menos populares para reducir el costo y la tarifa de transferencia.

https://docs.aws.amazon.com/whitepapers/latest/aws-fault-isolation-boundaries/points-of-presence.html

Redes de tier 1: es una red que puede alcanzar cualquier otra red en internet sin comprar tránsito de IP o pagar por "peering".
Las AZs de AWS están todas conectadas de manera redundante a múltiples proveedores de tránsito tier-1.

Servicios de AWS que usan PoPs para envío de contenido o subida acelerada:
* Amazon CloudFront. Es un CDN. Apuntás tu sitio web a CloudFront para que encamine las solicitudes a la cache de la edge location más cercana.
Te permite elegir un origen (como un servidor web) que será la fuente de lo cacheado.
Cachea el contenido de lo que retorna el origen a varias edge locations alrededor del mundo.
* Amazon S3 transfer acceleration: permite generar un URL especial que puede ser usada por usuarios finales para subir archivos a una edge location cercana. Una vez que un archivo es subido a una edge location, puede moverse mucho más rápido en la red de AWS para alcanzar S3.
* AWS global accelerator: puede encontrar el camino óptimo desde un usuario a tus servidores web. Global accelerator son desplegados en edge locations para que puedas tráfico de usuario a una edge location en lugar de hacerlo directamente a tu aplicación web.

AWS direct connect
Es una conexión privada/dedicada entre tu datacenter, oficina, co-ubicación y AWS.
¿Qué es una co-ubicación o co-location? es un datacenter donde el equipamiento, el espacio y  el ancho de banda está disponible para alquiler para clientes minoristas.
Tiene dos opciones de conexión de red muy rápidas:
* Ancho de banda más bajo: 50MBps-500MBps
* Ancho de banda más alto: 1GBps-10GBps
Ayuda a reducir costos de red e incrementa el rendimiento del ancho de banda.
Provee una experiencia de red más consistente que una típica conexión a internet (confiable y segura).

Direct Connect Locations
Son datacenters confiables asociados a los cuales podés establecer una conexión dedicada de alta velocidad y baja latencia desde tu "on-premise" a AWS.
Tendrías que usar AWS direct connect para establecer una conexión.

Local Zones
Son datacenters ubicados cerca de áreas densamente pobladas para proveer rendimiento de muy baja latencia para esa área (milisegundos de un dígito de latencia como, por ejemplo 7ms).
Los Angeles California fue la primera Local zone en ser desplegada.
Solo algunos servicios específicos se han hecho disponibles:
* Instancias tipo EC2 (T3, C5, R5, R5d, I3en, G4)
* EBS (io1 y gp2)
* Amazon FSx
* Application Load Balancer
* Amazon VPC
El propósito de las Local zones es apoyar aplicaciones de alta demanda sensibles a latencias:
* Medios y entretenimiento
* Automatoizaciòn de diseño electrónico
* Ad-tech
* Machine learning

WaveLength zones
Permiten "edge-computing" en redes 5G.
Las aplicaciones tendrán ultra baja latencia.
AWS se ha asociado con varias comapañìas para utilizar redes 5G (Verizon, Vodafone, SK telecom, etc).
Creàs una subred atada a una weblength zone y luego podès lanzar máquinas virtuales al borde de las redes 5G objetivo.

Data residency o residencia de datos
Es la ubicación geográfica en donde residen los recursos de una organización o nube.
Compliance boundaries o lìmites de cumplimiento: es un requerimiento legal por un gobierno u organización que describe a donde se les permite residir a los datos y recursos de la nube.
Soberanìa de datos: es el control jurisdiccional o autoridad legal que puede ser afirmada sobre datos porque su ubicación física se encuentra en límites jurisdiccionales.
Para cargas de traqbajo que necesitan cumplir con los lìmites de cumplimiento estrictamente definiendo la residencia de datos y recursos de la nube en AWS se puede usar:
* AWS config es una política como servicio de código. Podès crear reglas para revisar continuamente la configuración de los recursos de AWS.
* AWS outposts es un estante físico de servidores que podès poner en tu datacenter. Tus datos van a residir en donde resida fìsicamente tu outpost.
* IAM policies

AWS puede utilizarse para el sector público u organizaciones desarrollando cargas de trabajo en la nube para el sector público.
AWS logra esto por programas de cumplimiento regulatorio junto con controles de seguridad y gobernanza.
AWS tiene regiones especiales para regulaciones de Estados Unidos llamadas GovCloud.

AWS Ground Station
Es un servicio completamente gestionado que te permite controlar comunicaciones satelitales, procesar datos y escalar tus operaciones sin tener que preocuparte por construir o gestionar tu propia infraestructura de estación terrestre.
Casos de uso:

AWS Outposts
Es un servicio completamente gestionado que ofrece la misma infraestructura, servicios AWS , API y herramientas para cualquier datacenter, co-ubicación o instalación "on-premise" para una experiencia híbrida consistente.
AWS outposts es un estante de servidores ejecutando infraestructura de AWS en tu ubicación física.
Un estante de servidores es un marco diseñado para organizar equipo IT, ejemplo 42U Rack. La U es por "rack units" o "U spaces" que es igual a 1.75 pulgadas. El estándar de la industria es 48U, que serían siete pies.

Cloud architecture o arquitectura en la nube
---------------------------------------------------------------------------------------------------

¿Qué es un "solutions architect"?
Es un rol en una organización técnica que diseña una solución técnica usando múltiples sistemas vía búsqueda, documentación y experimentación.
¿Qué es un "cloud architect"?
Es un solutions archictect que está enfocado únicamente en soluciones técnicas arquitectónicas usando servicios cloud.
Un cloud architect necesita entender los siguientes términos y factorizarlos en su arquitectura basado en los requerimientos del negocio:
* Disponibilidad: la habilidad para asegurar que el servicio esté disponible.
* Escalabilidad: la habilidad pata crecer rápido o sin obstáculos.
* Elasticidad: la habilidad para creecer o encogerse para cumplir con la demanda.
* Tolerancia a fallos: la habilidad para prevenir fallos.
* Recuperación de desastre: la habilidad para recuperarse de los fallos.
Un solutions architect necesita siempre considerar los siguinetes factores de negocio:
* Seguridad: cuan segura es la solución.
* Costo: cuanto va a costar.

Alta disponibilidad
Es la habilidad de que un servicio permanezca disponible asegurando que no hayan ningún punto de fallos y asegurar cierto nivel de desampeño.
Asegurar tus cargas de trabajo a través de múltiples AZs asegura que si una o dos AZs dejan de estar disponibles, el servicio o aplicaciones permanezca disponible.
Elastic Load Balancer: Un load balancer o balanceador de carga permite distribuir el tráfico a múltiples servidores en uno o más datacenter. Si un datacenter o servidor deja de estar disponible, el load balancer va a enrutar el tráfico solo a datacenters disponibles con servidores.

Alta escalabilidad
Es la habilidad para incrementar la capacidad basado en el incremento de la demanda de tráfico, memoria y poder de cómputo.
* Escalado vertical (scaling up): mejorar a un servidor más grande.
* Escalado horizontal (scaling out): añadir más servidores del mismo tamaño.

Alta elasticidad
Es la habilidad para incrementar o decrementar automáticamente la capacidad basado en el tráfico actual, memoria y poder de cómputo. Se suele confundir con alta escalabilidad.
Escalado hotizontal:
* Scaling out: añadir más servidores del mismo tamaño.
* Scaling in: quitar servidores poco utilizados del mismo tamaño.
El escalado vertical es generalmente difícil para la arquitectura tradicional por lo que generalmente verás solamente escalado hotrizontal descrito con elasticidad.
Auto Scaling Groups (ASG): Es una característica de AWS que añadirá automáticamente o quitará servidores basado en regalas de escalado que definís basado en métricas.

Tolerancia a fallos
Es la habilidad de tu servicio de asegurar que no haya ningún punto de fallo.
Fail-over: es cuando tenés que planear cambiar el tráfico a un sistema redundante en caso de que el sistema principal falle. Un ejemplo común es tener una copia de tu base de datos en donde los cambios realizados se sincronizan. El sistema secundario no está en uso hasta que ocurre un "fail-over" y se vuelve la base de datos principal.
RDS Multi-AZ es cuando "corrés" una base de datos duplicada en stand by en otra zona de disponibilidad en caso de que tu base de datos principal falle.

Alta durabilidad
Es la habilidad para recuperarse de un desastre y prevenir la pérdida de datos. Las soluciones que recuperan de un desastre se conocen como "disaster recovery" (DR)
* ¿Tenés un backup?
* ¿Qué ten rápido podés recuperar ese backup?
* ¿Tu backup todavía funciona?
* ¿Cómo te asegurás de que los datos actuales no está corruptos?
CloudEndure Disaster Recovery replica continuamente tus máquinas a áreas de "staging" de bajo costo en tu cuenta objetivo de AWS y región preferida posibilitando una recuperación rápida y confiable en caso de fallos en el datacenter.

Business continuity plan (BCP)
Es un documento que describe como un negocio continuará operando durante una ruptura no planeada de los servicios.
* Recovery Point Objective (RPO): Es el máximo aceptable de pérdida de datos después de un incidente no planificado, expresado como una cantidad de tiempo.
* Recovery Time Objective (RTO): Es el máximo tiempo de inactividad que tu negocio puede tolerar sin incurrir en una pérdida financiera significativa.

Hay múltiples opciones para recuperación que cambian costo vs tiempo para recuperar.
* Backup y recuperación: se hace un backup de tus datos a una nueva infraestructura. RPO/RTO -> horas.
* "Pilot light": Los datos son replicados a otra región con los mínimos servicios ejecutándose. RPO/RTO -> 10 minutos.
* "Warm standby": copia escalada para abajo de tu infraestructura ejecutándose lista para escalar para arriba. RPO/RTO -> minutos.
* "Multi-site Active/active": copia escalada para arriba de tu infraestructura en otra región. RPO/RTO -> tiempo real.

https://media.geeksforgeeks.org/wp-content/uploads/20250419125835918183/AWS-Disaster-Recovery.webp

Recovery time objective (RTO): es el máximo retraso aceptable entre la interrupción del servicio y la recuperación del mismo. Este objetivo determina la que es considerada una ventana aceptable de tiempo cuando el servicio no está disponible y es definido por la organización.

Recovery point objective (RPO): Es la máxima cantidad de tiempo aceptable desde el último punto de recuperación de datos. Este objetivo determina lo que es considerado una aceptable pérdida de datos entre el último punto de recuperación y la interrupción del servicio y es definido por la organización.

Herramientas de gestión y desarrollo
------------------------------------------------------------------------------------------------------
API de AWS
Una API es un software que permite que dos aplicaciones o servicios se comuniquen entre sí. El tipo más común es vía solicitudes HTTP/S
La API de AWS es una API HTTP con la cual se puede interactuar enviando solicitudes HTTP usando aplicaciones como Postman.
Cada servicio de AWS tiene su propio endpoint al cual podés enviarle solicitudes.
Es raro que los usuarios envíen solicitudes directamente a la API de AWS. Es mucho más fácil interactuar con la API usando herramientas de desarrollador: AWS management console; AWS SDK, interactuá con la API usando tu lenguaje de programación favorito; AWS CLI, interactuá con la API usando una Shell/terminal.

AWS management console

Modelo de responsabilidad compartida
-------------------------------------------------------------------------------------------------------
Modelo de responsabilidad compartida
Es un marco de trabajo de la nube que define las obligaciones de seguridad del cliente versus el CSP, ejemplo: AWS
https://d1.awsstatic.com/onedam/marketing-channels/website/aws/en_US/product-categories/security-identity-compliance/compliance/approved/images/7a404923-5572-409c-b30e-6d44706bcd89.094727e5c591e9a96edf10578d0bc1172d9e4553.jpeg
Cada CSP tiene su propia variante de modelo de responsabilidad compartida pero son generalmente los mismo.

Modelo de responsabilidad compartida de AWS
* AWS es responsable de todo lo físico: hardware e infraestructura global -> regiones , zonas de disponibilidad, Edge locations, seguridad física
Con respecto al software, AWS es responsable del almacenamiento, bases de datos, redes, cómputo.
* El cliente es responsable de la configuración de los servicios gestionados de software de terceros: plataformas, aplicaciones, gestión de acceso e identidad (IAM).
El cliente es también responsable de la configuración de la infraestructura virtual y los sistemas: elegir el sistema operativo, redes, firewall
El cliente es también responsable de la encriptación de datos del lado del cliente, encriptación del lado del servidor, protección del tráfico de redes, datos del cliente.

Los clientes son responsables de la seguridad EN la nube.
AWS es responsable de la seguridad DE la nube.

Tipos de responsabilidad en computación en la nube
* On-premise: el cliente es responsable de todo
* Infrastructure as a service: el cliente es responsable de las aplicaciones, datos, runtime, middleware y sistema operativo. El CSP es responsable de virtualización, servidores, almacenamiento y redes.
* Platform as a service: El cliente es responsable de las aplicaciones y los datos. El CSP de todo lo demás.
* Software as a service: El CSP es responsable de todo.

Ejemplo de comparación de modelo de responsabilidad compartida
* IaaS:
 - Bare Metal , EC2 Bare metal instance: el cliente es responsable del sistema operativo anfitrión e hipervisor. AWS es responsable de la máquina física
 - Máquina virtual , Elastic Cloud Compute (EC2): el cliente es responsable de la configuración del sistema operativo huésped, del container runtime (ejemplo: docker). AWS es responsable de hipervisor, de la máquina física y del sistema operativo anfitrión.
 - Contenedores , AWS Elastic Container Service (ECS): el cliente es responsable de la configuración de los contenedores, despliegue de contenedores, almacenamiento de contenedores. AWS es responsable de el sistema operativo (huésped y anfitrión), el hipervisor y del container runtime.
*PaaS:
 - Plataforma gestionada , AWS Elastic Beanstalk: el cliente es responsable de subir el código, algo de configuración del entorno, estrategias de despliegue, configuración de servicios asociados. AWS será responsable de los servidores, sistema operativo, redes, almacenamiento, seguridad.
*SaaS:
 - Colaboración de contenido , Amazon WorkDocs: el cliente es responsable del contenido de los documentos, gestión de archivos, configuración de los controles de acceso compartido. AWS será responsable de los servidores, sistema operativo, redes, almacenamiento y seguridad.
*FaaS (functions as a service):
 - Funciones, AWS lambda: el cliente es responsable de subir el código. AWS será responsable de todo lo demás.

Elastic compute cloud (EC2)
Te permite lanzar máquinas virtuales. Una máquina virtual es una emulación de una computadora física usando software.
La virtualización del servidor te permite crear, copiar, cambiar de tamaño o migrar tu servidor.
Múltiples máquinas virtuales pueden ejecutarse en el mismo servidor físico para compartir el costo con otros clientes.
Cuando lanzamos una VM (máquina virtual) la llamamos instancia.
EC2 es un servidor altamente configurable donde podés elegir AMI (amazon machine image) que afecta las siguientes opciones:
* La cantidad de CPUs
* La cantidad de RAM
* La cantidad de ancho de banda
* El sistema operativo
* Unir múltiples discos duros virtuales	para almacenamiento, ejemplo: Elastic Block Store (EBS)

AMI o amazon machine image
Es una configuración predefinida para una máquina virtual

EC2 es también considerado la columna vertebral de AWS porque la mayoría de los servicios de AWS están usando EC2 como sus servidores subyacentes, ejemplos: S3, RDS, DynamoDB, Lambdas

Amazon LightSail
Es el servicio de servidores virtuales gestionados. Es la versión "amigable" de las máquinas virtuales de EC2. Cuando querés lanzar un servidor de Windows o Linux pero no tenés mucho conocimiento de AWS. Ejemplo, lanzar un WordPress.

Contenedores: virtualizar un sistema operativo para ejecutar múltiples cargas de trabajo en una sola instancia de un sistema operativo. Los contenedores se usan generalmente en arquitectura de microservicios
Elastic Container Service (ECS)
es un servicio de orquestación de contenedores que soporta contenedores de Docker. Lanza un cluster de servidores en instancias EC2 con Docker instalado. Es para cuando necesitás Docker como servicio o necesitás ejecutar contenedores.
Elastic Container Regisitry (ECR)
Es un repositorio para imágenes de contenedores. Para lanzar contenedores necesitás una imagen. Un repositorio es un almacenamiento que tiene control de versiones.
ECS Fargate
Es un servicio de orquestación de contenedores serveless. Es como ECS, pagás on-demand por cada contenedor en ejecución (con ECS tenés que mantener un servidor EC2 corriendo incluso si no tenés contendores ejecutándose). AWS gestiona el servidor subyacente para que no tengas que escalar o mejorar el servidor EC2.
Elastic Kubernetes Service (EKS)
Es un servicio de kubernetes completamente gestionado. Kubernetes (K8) es un software de código abierto que fué creado por Google y es generalmente un estándar para gestión de microsevicios. Es para cuando necesitás ejecutar kubernetes como servicio.

Serverless: es cuando los servidores subyacentes son gestionados por AWS. No debés preocuparte por configurar los servidores.
AWS Lambda
Es un servicio de funciones serverless. Podés ejecutar código sin gestionar servidores. Subís pequeñas piezas de código, elegís la cantidad de memoria y cuanto tiempo se le permite a la función ejecutarse antes de que se agote el tiempo. Pagás de acuerdo al tiempo de ejecución de la función serverless redondeado a los 100ms más cercanos.

Servicios de computación de alto rendimiento

The Nitro System: Una combinación de hardware dedicado e hipervisor liviano que permite innovación más rápida y seguridad mejorada. Todas los tipos de instancias EC2 usan el Nitro System
* Nitro cards: tarjetas especializadas para VPC, EBS y tarjeta controladora.
* Nitro security chips: integrados en la placa madre. Protege los recursos de hardware.
* Nitro hypervisor: memoria de hipervisor liviana y rendimiento tipo bare metal de asignación de CPU.

Instancias bare metal: Podés lanzar instancias EC2 sin hypervisor para que puedas ejecutar cargas de trabajo directamente en el hardware para máximo rendimiento y control. Las instancias EC2 M5 y R5 se ejecutan como bare metal.

Bottlerocket: es un sistema operativo de código abierto basado en Linux cuyo propósito es ejecutar contenedores en máquinas virtuales o anfitriones bare metal.

Computación de alto rendimiento o HPC: es un cluster de cientos de miles de servidores con conexiones rápidas entre ellos con el propósito de aumentar la capacidad computacional. Cuando necesitás una super computadora para llevar a cabo problemas computacionales muy grandes para ejecutarse en computadoras estándar o que tomarían mucho tiempo.

AWS ParallelCluster es una herramienta de gestión de clusters de código abierto soportada por AWS que hace que sea fácil el desplegar y gestionar clusters HPC en AWS.

Edge computing y hybrid comnuting

* Edge computing: es cuando empujàs tus cargas de trabajo de computación fuera de tus redes para correr cerca de las ubicación destino. Ejemplo: empujar para correr en teléfonos, dispositivos IoT o servidores externos dentro de tu red de la nube.
* Hybrid computing: es cuando podès correr tus cargas de trabajo tanto en tu datacenter on-premise como en tu AWS VPC.

AWS Local Zones: son Edge datacenters localizados afuera de una región de AWS para que puedas usar AWS cerca de tu destino. Cuando necesitàs poder de còmputo màs rápido, almacenamiento y bases de datos en áreas muy pobladas que están fuera de una región de AWS.

Tipos de servicios de almacenamiento

Elastic Block Store (EBS) - Block
Los datos son partidos en bloques
Directamente accedido por el sistema operativo
Soporta un solo volumen de escritura
Protocolos: FC, iSCSI
Es para cuando necesitás un disco rígido virtual unido a una máquina virtual

AWS Elastic File Storage (EFS) - File
El archivo es almacenado con datos y metadatos
Múltiples conexiones vía comunicación de red
Soporta múltiples lecturas, la escritura bloquea el archivo
Protocolos: NFS, SMB
Cuando necesitás compartir archivos donde múltiples usuarios o máquinas virtuales necesitan acceder a la misma unidad de almacenamiento

El objeto es almacenado con datos, metadatos y un único ID
Soporta múltiples lecturas y escrituras (sin bloqueos)
Protocolos: HTTP/S, API
Cuando necesitás subir archivos y no tener que preocuparte por la infraestructura subyacente. No está pensado para cuando hay muchos inputs y outputs por segundo

Introducción a S3
Almacenamiento basado en objetos: Es una arquitectura de almacenamiento de datos que gestiona los datos como objetos, a diferencia de otras arquitecturas de almacenamiento.
En otras arquitecturas de almacenamiento:
* Los sistemas de archivos gestionan datos como un archivo y una jerarquía
* Almacenamiento en bloque que gestiona los datos como bloques con sectores y rastros

S3 te provee de almacenamiento ilimitado, no necesitás tener en cuenta la infraestructura subyacente. La consola de S3 te provee de una interface para que puedas subir datos y acceder a ellos.

Objetos S3
* Los objetos contienen tus datos, son como archivos
* Un objeto consiste en:
 - Una clave: es el nombre del objeto
 - Un valor: los datos propiamente dichos compuestos de una secuencia de bytes
 - Versión ID: la versión del objeto
 - Metadatos: información adicional unida al objeto

S3 Bucket
* Los buckets guardan objetos. Pueden también pueden tener carpetas que guardan objetos.
* S3 es un namespace universal, por lo que un bucket debe tener un nombre único (parecido a tener un nombre de dominio)

Cada objeto puede tener de 0 B a 5 TB en tamaño

---
Clases de almacenamiento S3

AWS ofrece un rango de clases de almacenamiento S3 que intercambia tiempo de recuperación, accesibilidad y durabilidad por precios más bajos.
* S3 standard: es el más caro y es el que está por defecto. Es rápido, tiene 99.99% de disponibilidad. Replicado a través de tres zonas de disponibilidad.
* S3 intelligent tearing: usa ML para analizar el uso de objetos y determinar la clase de almacenamiento apropiada. Los datos son movidos al tier de acceso con mejor costo, sin ningún impacto en el rendimiento.
* S3 standard-IA (Infrequent access): es bastante rápido pero es más barato si accedés a tus archivos menos de una vez al mes. Una tarifa adicional de recuperación es aplicada. Tiene disponibilidad reducida.
* S3 One-Zone-IA: bastante rápido. Los objetos solo existen en una zona de disponibilidad. Las disponibilidad es de 99.5%. Es 20% más barato que standard-IA
* S3 Glacier: para un almacenamiento frío de largo plazo. La recuperación de datos puede tomar entre minutos y horas pero es muy barato.
* S3 Glacier Deep Archive: es el de menor costo. La recuperación de datos lleva 12 horas.

AWS snow family
Son dispositivos de almacenamiento y computación para mover físicamente datos hacia la nube o desde la nube cuando el movimiento de datos por internet o por conexiones privadas es lento, costoso o difícil.
Los dispositivos son: Snowcone, Snowball Edge y Sonwmobile
* Snowcone: viene en dos tamaños: HDD de 8TB y SSD de 14TB
* Snowball Edge: viene generalmente en dos tipos: almacenamiento optimizado (80TB) y computación optimizada (39.5 TB)
* Sonwmobile: 100PB de almacenamiento

Los datos son enviados a Amazon S3

Servicios de almacenamiento
* Simple Storage Service (S3): es un servicio de almacenamiento de objetos serverless. Podés subir archivos muy grandes y una cantidad ilimitada de archivos. No debés preocuparte por el sistema de archivos subyacente, o mejorar el tamaño del disco.
* S3 Glacier: es de muy bajo costo y está pensado para el largo plazo (back up). Utiliza HDDs para tener bajo costo. Es muy seguro y durable.
* Elastic Block Store (EBS): es un servicio de almacenamiento bloque persistente. Es un disco duro virtual que unís a instancias EC2. Podés elegir diferentes tipos de discos duros: SSD, IOPS SSD, HDD de rendimiento, HDD frío
* Elastic File Storage (EFS): es un servicio de sistema de archivos NFS cloud-native. Almacenamiento de archivos que podés montar en múltiples instancias EC2 al mismo tiempo. Cuando necesitás compartir archivos entre múltiples servidores.
* Stoarge Gateway: es un servicio de almacenamiento en la nube híbrido que extiende tu almacenamiento on-premise a la nube. Hay tres opciones: File Gateway, extiende tu almacenamiento local a AWS S3; Volume Gateway, cachea tus unidades locales a S3 para que puedas tener backup de archivos locales en la nube; Tape Gateway, guarda archivos en cintas virtuales para hacer back up de tus archivos a un buen costo en el largo plazo.
* AWS snow family: son dispositivos usados para migrar físicamente grandes cantidades de datos a la nube.
 - Snowball y Snowball Edge son dispositivos de almacenamiento del tamaño de un maletín con una capacidad de 50 a 80 TB.
 - Snowmobile es un contenedor lleno con estantes de almacenamiento y computación que es transportado vía un camión para transferir 100PB de datos por trailer.
 - Snowcone es una versión muy pequeña de snowball que puede transferir 8 TB de datos.
* AWS Backup es un servicio de backup completamente gestionado que hace que sea fácil centralizar y automatizar el backup de los datos a través de múltiples servicios de AWS, por ejemplo: EC2, EBS, RDS, DynamoDB, EFS, Storage Gateway. Podés crear planes de backup.
* CloudEndure Disaster Recovery, continuamente replica tus máquinas en un área de staging de bajo costo en tu cuenta objetivo de AWS y región preferida, permitiendo una recuperación rápida y confiable en caso de fallas de datacenters.
* Amazon FSx es un sistema de archivos de alto rendimiento que puede ser usado en Windows (SMB) o Linux (Lustre)
 - Amazon FSx para Windows File Server usa el protocolo SMB y te permite montar FSx a servidores de Windows.
 - Amazon FSx for Lustre usa el sistema de archivos de Linux Lustre y te  permite montar FSx a servidores Linux.

Bases de datos
----------------------------------------------------------------------------------------------
¿Qué es una base de datos?
Una base de datos es un almacenamiento de datos que guarda datos estructurados y semi-estructurados
Una base de datos es un almacenamiento de datos más complejo porque requiere el uso de un diseño formal y técnicas de modelado.

Las bases de datos pueden ser relacionales: datos estructurados que representan datos de manera tabular (tablas, filas y columnas)
Bases de datos no relacionales: semi-estructuradas que pueden o no utilizar datos de manera tabular

----------
¿Qué es un depósito de datos? (data warehouse)
Es un almacenamiento de datos diseñado para cargas de trabajo analíticas, que es generalmente orientado en columnas.
Las compañías tendrán terabytes y millones de filas de datos, y necesitan una manera rápida de producir reportes analíticos.
Los depósitos de datos generalmente llevan a cabo agregado (aggregation)
El agregado es agrupar datos, por ejemplo, encontrar un total o un promedio.
Los depósitos de datos están optimizados alrededor de columnas ya que necesitan rápidamente agregar datos de columnas

Los depósitos de datos están diseñados para ser "calientes". Con calientes se refiere a que pueden retornar consultas muy rápido aunque tengan muchísimos datos.

Los depósitos de datos son accedidos de manera poco frecuente, lo que significa que no están diseñados para reportes en tiempo real pero tal vez una o dos veces en un día o una vez a la semana para generar negocios y reportes de usuario.

Un depósito de datos necesita consumir datos de bases de datos relacionales de forma regular.

---------
¿Qué es un almacenamiento clave valor? (key value store)
Es un tipo de base de datos no relacional (NoSQL) que usa un método de clave y valor para almacenar datos. Debido a su diseño simple pueden escalar muy bien.

--------
¿Qué es una base de datos de documentos? (document database)
Es una base de datos NoSQL que almacena documentos como su principal estructura de datos. Un documento podría ser un XML pero más comúnmente un JSON o algo similar. Los almacenamientos por documentos son subclases de almacenamiento clave/valor.
Los componentes de un documento son comparables a una base de datos relacional:

SQL   |  Documento
-------------------
tablas   | Colección
filas    | Documentos
columnas | Campos
índices  | índices
"joins"  | "embedding" o "linking"

-------------
Servicios de bases de datos NoSQL
* DynamoDB: es una base de datos NoSQL de documentos y pares clave valor serverless. Está diseñada para escalar miles de millones de registros con un retorno de datos consistente garantizado en por lo menos un segundo. No tenés que preocuparte por gestionar fragmentos.
DynamoDB es el buque insignia de los servicios de bases de datos de AWS lo que significa que siempre que pensamos en un servicio de base de datos que escale, tenga buen costo y muy rápido deberíamos pensar en DynamoDB
* DocumentDB: es una base de datos NoSQL basada en documentos que es "compatible con MongoDB". Hubieron problemas de licencia open-source al usar MongoDB, por lo que AWS decidió crear su propia base de datos tipo MongoDB
* Amazon Keyspaces: es una base de datos Apache Cassandra completamente gestionada. Cassandra es una base de datos open-source NoSQL de clave y valor similar a DynamoDB en que es en columnas pdero tiene funciones adicionales.

--------------
Servicios de bases de datos relacionales
* Relational Database Service (RDS): es un servicio de bases de datos relacional que soporta múltiples motores SQL. Relacional es sinínimo de SQL y procesamiento transaccional en línea (OLTP)
RDS soporta los siguientes motores de bases de datos: MySQL, MariaDB, Postgres, Oracle, Microsoft SQL server, Aurora.
* Aurora: es una base de datos completamente gestionada de MySQL (5 veces más rápida) y PSQL (3 veces más rápida). Es para cuando querés una base de datos relacional disponible, durable, escalable y segura para Postgres o MySQL.
* Aurora Serverless: es la versión serverless on-demand de Aurora. Es para cuando querés la mayor parte de los beneficios de Aurora pero podés ceder "cold-starts" o no tenés mucha demanda de tráfico
* RDS on VMware: te permite desplegar motores soporados por RDS a un datacenter om-premise. El datacenter debe usar VMware para virtualización del servidor. Es para cuando querés bases de datos gestionadas por RDS en tu propio datacenter.

------------
Otros servicios de bases de datos
* Redshift es un warehouse de datos de petabytes de tamaño. Los warehouse de datos pueden ser caras porque mantienen a los datos "calientes", lo que significa que puede ejecutar una consulta compleja y una gran cantidad de datos y recoger los datos de vuelta de manera muy rápida. Es para cuando generás reportes o analíticas de una gran cantidad de datos.
* ElasticCache es una base de datos gestionada de las bases de datos en memoria y de cache open-source Redis o Memcached. Es para cuando querés mejorar el rendimiento de aplicación al añadir una capa de cache en frente del servidor web o la base de datos.
* Neptune: es una base de datos gestionada de grafos. Los datos son representados como nodos interconectados. Es para cuando necesitás entender las conexiones entre datos, por ejemplo, mapear relaciones en redes sociales
* Amazon Timestreams: se puede pensar como un dispositivo que envía un montón de datos que son sensibles al tiempo como dispositivos IoT. Es para cuando querés medir como las cosas cambian con el tiempo.
* Amazon Quantum Ledger Database: es una base de datos que provee logs de transacciones variables transparentes, inmutables y criptográficos. Es para cuando necesitás registrar el historial de actividades financieras que son confiables.
* Database Migration Service (DMS): es un servicio de migración de bases de datos. Podés migrar de:
 - bases de datos on-premise a AWS
 - desde dos bases de datos diferentes o las mismas cuentas de AWS usando diferentes motores de SQL
 - desde SQL a NoSQL

Redes (networking)
------------------------------------------------------------------------------------------------
Servicios de redes cloud-native

VPC: es una sección lógicamente aislada de la nube de AWS donde podés lanzar recursos de AWS.
Internet Gateway: permite el acceso a internet
Route tables: determinan donde el tráfico de red de tus subredes es direccionado
Región: la ubicación geográfica de tu red
AZ: el data center de tus recursos de AWS
Subredes o subnets: una partición lógica de una red IP en múltiples segmentos de red más pequeños.
NACLs: funcionan como firewalls al nivel de la subred
Security groups: funcionan como firewall a nivel de instancia

------------
Servicios de red empresariales/híbridos

* AWS Virtual Private Network (VPN): una conexión segura entre on-prmise, oficinas remotas y empleados móviles.
* DirectConnect: conexión dedicada de gigabits de un data center on-premise a AWS (conexión muy rápida)
* PrivateLinks (VPC interface ebdpoints): mantiene el tráfico dentro de la red AWS y no atraviesa el internet para mantener el tráfico seguro.

------------
Virtual private cloud (VPC) y subredes

Virtual Private Network (VPC): es una sección lógicamente aislada de la red de AWS a domde podés lanzar tus recursos de AWS. Elegís un rango de IPs usando rango CIDR

Las subredes necesitan tener un rango más pequeño de CIDR que el VPC

Subred pública: es una red que puede alcanzar internet

Subred privada: es una red que no puede alcanzar internet

---------
Security groups vs NACLs

NACLs (network Access control lists): funciona como un firewall virtual a nivel de subred. Podés crear reglas sobre lo que se permite y lo que se denega, por ejemplo: bloquear una dirección IP específica por abuso.

Security groups: Funcionan como firewalls a nivel instancia. Deniega todo el tráfico de manera implícita. Solo creas reglas sobre lo que se permite, por ejemplos: permitir que una instancia EC2 acceda al puerto 22 para SSH; no podés bloquear ninguna dirección IP

EC2
---------------------------------------------------------------------------------------------------
Introducción a EC2

Elastic Compute Cloud (EC2) es un servidor virtual altamente configurable
EC2 es capacidad computacional redimensionable. Toma minutos lanzar nuevas instancias.
Todo en AWS usa instancias EC2 por debajo.

* Se puede elegir el sistema operativo via AMI (Amazon Mchine Image): Red Hat, Ubuntu, Windows, Amazon Linux, Suse
* Elegir el tipo de instancia: t2.nano o C4.8xlarge (hay diferencias en las especificaciones técnicas)
* Añadir almacenamiento (EBS, EFS)
  - Si se elige EBS hay que elegir entra las siguientes opciones: SSD, HDD, Virtual magnetic tape, Multiple volumes
* Configurar la instancia: Security Groups, Key Pairs, UserData, IAM Roles, Placement Groups

-------------
Familias de instancias EC2

Las familias de instancia son diferentes combinaciones de CPU, memoria, almacenamiento y capacidad de red.
Las familias de instancias te permiten elegir la combinación apropiada de capacidad para satisfacer los requerimientos únicos de tu aplicación.
Diferentes familias de instancias son diferentes por las variaciones de hardware usadas para darles sus propiedades únicas.

Tipos de familias de instancias
* Propósito general: A1, T2, T3, T3a, T4g, M4, M5, M5a, M5n, M6zn, M6g, M6i, Mac. Tienen un buen balance de cómputo, memoria y recursos de red. Casos de uso: servidores web y repositorios de código.
* Compute optimized: C5, C4, Cba, C5n, C6g, C6gn. Ideales para aplicaciones que necesitan alto rendimiento de procesador. Casos de uso: modelado científico, servidores de juegos dedicados
* Memory optimized: R4, R5, R5a, R5b, X1, X1e, High memory, z1d. Alto rendimiento para cargas de trabajo que procesan grandes conjuntos de datos en memoria. Casos de uso: caches en memoria, bases de datos en memoria, big data analytics en tiempo real
* Accelerated optimized: P2, P3, P4, G3, G4ad, G4dn, F1, inf1, VT1. Aceleradores de hardware o co-procesadores. Casos de uso: Machine learning, finanzas computacionales, análisis sísmico, speech recognition.
* Storage optimized: I3, I3en, D2, D3, D3en, H1. Lectura secuencial y acceso de escritura a grandes conjuntos de datos en almacenamiento local. Casos de uso: NoSQL, bases de datos transaccionales o en memoria, data warehousing

Un instance type es una combinación entre tamaño y familia.
	
-------------------
Tipos de instancias EC2

Un tipo de instancia o instance type es un particular tamaño de instancia y familia de instancia

Un patrón común de tamaños de instancia es el siguiente: nano, micro, small, medium, large, xlarge, 2xlarge, 4xlarge, 8xlarge. Ejemplo: t2.micro. Cabe aclarar que hay excepciones para este patrón de tamaños, por ejemplo: c6g.metal, es una máquina baremetal; C5.9xlarge

Los tamaños de las instancias EC2 generalmente doblan en precio	y atributos clave
Ejemplo:

nombre    |  vCPU  |  RAM(GIB)  |  on-demand por hora  |  on-demand por mes

t2.small     1         12               $0.023               $16.79
t2.medium    2         24               $0.0464              $33.87
t2.large     2         36               $0.0928              $67.74
t2.xlarge    4	       54               $0.1856              $135.48

------------------
Anfitriones dedicados vs instancias dedicadas

Los anfitriones dedicados son instancias EC2 inquilinas diseñadas para dejarte "Bring-Your-Own-Licence" (BYOL) basado en características de máquina

* Instancia dedicada
  - Aislamiento de instancia
  - Facturación por instancia
  - Visibilidad de características físicas: sin visibilidad
  - Afinidad entre un anfitrión y la instancia: sin afinidad
  - "Targeted instance placement": Sin control
  - Colocación automático de instancia: sí
  - Añadir capacidad usando una solicitud de asignación: no

* Anfitriones dedicados
  - Aislamiento físico de servidor
  - Facturación por anfitrión
  - Visibilidad de características físicas: Sockets, núcleos, id del anfitrión
  - Afinidad entre un anfitrión y la instancia: despliegue de consistencia a las mismas instancias en el mismo servidor
  - "Targeted instance placement": Control adicional sobre colocación de instancia en el servidor físico
  - Colocación automático de instancia: sí
  - Añadir capacidad usando una solicitud de asignación: sí

-------------
Tenencia de EC2

EC2 tiene tres niveles de tenencia

* Anfitrión dedicado: tenés control sobre los atributos físicos de tu servidor (el servidor entero)
* Instancia dedicada: tu servidor está en la misma máquina física que la de otros clientes
* Por defecto: tu instancia vivirá en algún lugar del servidor y cuando hagas reboot va a estar en otro lugar

-----------
Modelo de precios de EC2

Hay 5 maneras de pagar por EC2 (máquinas virtuales)
* On-demand:
  - flexible y de bajo costo
  - se paga por hora o por segundo
  - cargas de trabajo impredecibles y de corto plazo
  - no puede ser interrumpido
  - "for first time apps"
  - es el que requiere menos compromiso
* Spot hasta 90%:
  - "request spare computing capacity"
  - "flexible start and endtimes"
  - puede manejar interrupciones (si el servidor se detiene o empieza de manera aleatoria)
  - "for non-critical background jobs"
  - es con el que más se ahorra
* Reservado hasta 75% menos:
  - uso predecible y estado firme
  - comprometerte con EC2 en un plazo de 1 o 3 años
  - podés revender instancias reservadas no usadas
  - es el mejor para el largo plazo
* Dedicado:
  - servidores dedicados
  - puede ser on-demand, reservado o spot
  - cuando necesitás garantía de hardware aislado (requerimientos empresariales)
  - es el más caro

-----------------------------------------------------------------
Modelos de precios de EC2
-----------------------------------------------------------------
On-demand

Es un modelo PAYG (pay as you go), donde consumís cómputo y luego pagás

Cuando lanzás una instancia EC2 está usando precio on-demand por defecto

No tiene pago "up-front" ni compromiso a largo plazo

Pagás por el segundo (mínimo 60 segundos) o por la hora

por segundo para:
* Linux
* Windows
* Windows con SQL Enterprise
* Windows con SQL estándar
* Windows con instancias web SQL que no tienen cobro de horas por separado

por hora para:
* Para todos los tipos de instancias

On-demand es para aplicaciones a donde la carga de trabajo es para corto plazo o impredecible
Es para cuando tenés una nueva app para desarrollo o querés experimentar

-----------------
Reservado, instancias reservadas (RI)

Diseñado para aplicaciones que tienen un estado firme, uso predecible o requieren capacidad reservada

El precio reducido se basa en: Term x Class offering x RI attributes x Payment option

Term: mientras más largo sea el plazo más grande es el ahorro. Te comprometés a un contrato de 1 o 3 años. Las instancias reservadas no se renuevan automáticamente. Cuando expire tu instancia va a usar on-demand sin interrupción del servicio

Class offering: mientras menos flexible más se ahorra.
* Estándar: hasta 75% de reducción de precio comparado con on-demand; podés modificar atributos RI
* Covertible: hasta 54% de precio reducido comparado con on-demand. Podés intercambiar RI basado en atributos RI si es más grande o igual en valor
* Programado: AWS ya no ofrece RI programado

Opciones de pago: mientras más se pague por adelantado más se ahorra
* "All upfront": el pago completo se realiza al principio
* "Partial upfront": una porción del costo se paga por adelantado y las horas restantes del período se facturan a una tasa de descuento por hora
* "No upfront": se te factura a una tasa de descuento por hora por cada hora dentro del período, a pesar de que la instancia reservada está siendo usada.

Las RIs (reserved instances) pueden ser compartidos entre múltiples cuentas dentro de una organización de AWS
Las RIs sin uso pueden ser vendidos en el Reserved Instance Marketplace


-----------------
Atributos RI

Atributos RI (AKA atributos de instancia) son limitados basados en class offering y pueden afectar el precio final de una RI. Hay cuatro atributos RI:
1. Tipo de instancia: por ejemplo , m4.large. Está conformado de la familia de la instancia (en este ejemplo m4) y el tamaños de la instancia (large en este caso)
2. Región: la región en la que la instancia reservada (RI) es comprada
3. Tenencia: si tu instancia corre en hardware compartido (por defecto) o dedicado
4. Plataforma: el sistema operativo, por ejemplo Windows o Linux

------------------
RI zonal y regional

Cuando comprás una RI, vos determinás el alcance de la instancia reservada.
El alcance no afecta al precio

RI regional: compra por región
* No reserva capacidad
* Descuentos de RI aplican al uso de instancia en cualquier AZ en la región
* Los descuentos RI aplican a uso de instancia dentro de la familia de instancia. Solo soportado en instancias reservadas Amazon Linux/Unix con tenencia por defecto.
* Podés hacer colas de compras para RI regional

RI zonal: compra por zona de disponibilidad (availability zone)
* Reserva capacidad en al AZ especificada
* Descuentos de RI aplican a instancia en la AZ seleccionada (no hay flexibilidad con las AZs)
* Sin flexibilidad en tamaño de instancia. Los descuentos RI aplican al uso de instancia solamente para el tipo de instancia especificado y el tamaño.
* No podés hacer colas para compras para RI zonal

--------------------
Límites RI

Hay un límite de instancias reservadas que podés comprar por mes

Por mes podés comprar:
* 20 instancias reservadas regionales por región
* 20 instancias reservadas zonales por AZ

Límites regionales
No podés exceder tu límite de instancias on-demand corriendo por comprar instancias reservadas regionales.
El límite por defecto de instancias on-demand es 20
Antes de comprar RI debés asegurarte de que tu límite on-demand sea igual o más grande que el RI que intentás comprar

Límites zonales
Podés exceder  tu límite de instancias on-demand coorriendo al comprar instancias reservadas zonales
Si ya tenés 20 instancias on-demand corriendo, y comprás 20 instancias reservadas zonales, podés lanzar  más de 20 instancias on-demand que coinciden con las especificaciones de tus instancias reservadas zonales.

------------------
Reservaciones de capacidad (Capacity reservation)

Las instancias EC2 son respaldadas por diferentes tipos de hardware, y por eso hay una cantidad finita de servidores disponibles dentro de una AZ por tipo de instancia o familia

Puede ocurrir que quieras lanzar un tipo específico de instancia EC2 pero a AWS se le hayan acabado los servidores

Capacity reservation es un servicio de EC2 que te permite solicitar una reserva de instancia EC2 para una región específica y AZ

La capacidad reservada es cobrada a la tasa on-demand del tipo de instancia seleccionada sin importar si la instancia está corriendo o no

Podés también usar tus instancias reservadas regionales con tus capacity reservations para beneficiarte de descuentos de facturación

---------------------
RI convertible vs RI estándar

RI estándar
* Los atributos RI pueden ser modificados:
  - cambiar la AZ dentro de la región
  - cambiar el alcance del RI zonal a RI regional y viceversa
  - cambiar el tamaño de instancia (Linux/UNIX solamente, tenencia por defecto)
  - cambia la red de EC2-classic a VPC y viceversa
* No se pueden intercambiar
* Puede ser comprado o vendido en el Marketplace de RI

RI convertible
* Los atributos RI no pueden ser modificados (llevás a cabo un intercambio)
* Puede ser intercambiado durante el plazo por otro RI convertible con nuevos atributos RI, incluyendo:
  - familia de instancia
  - tipo de instancia
  - plataforma
  - alcance
  - tenencia
* No puede ser comprado o vendido en el Marketplace RI

-----------------------
RI Marketplace

EC2 reserved instance matketplace te permite vender RI estándar sin usar para recuperar tu gasto de RI por RI que no usán o no podés usar

* Reserved instances pueden ser vendidas después de que han estado activas por al menos 30 días y una vez que AWS ha recibido pago por adelantado (si es aplicable)
* Deberás tener una cuenta de banco de Estados Unidos para vender instancias reservadas en el RI marketplace
* Debe haber por lo menos un mes restante en el plazo de la RI que estás listando
* Vas a conservar el precio y el beneficio de capacidad de tu reservación hasta que sea vendida y la transacción completada
* El nombre de tu compañía será compartido con el comprador para propósitos de impuestos
* Un vendedor puede solamente asignar el precio por adelantado para una RI. El precio de uso y otra configuración (por ejemplo, tipo de instancia, AZ, plataforma) va a permanecer igual a cuando la RI fue inicialmente comprada
* El tamaño del plazo será redondeado para abajo al mes más cercano. Por ejemplo, una reservación con 9 meses y 15 días restantes aparecerá como 9 meses en el Marketplace de RI
* Podés vender hasta $20000 en RI por año
* RIs en la región GovCloud no pueden ser vendidas en el RI marketplace

------------------------
Spot instances

AWS tiene capacidad de cómputo sin usar y quieren maximizar la utilidad de sus servidores inactivos. Es como cuando un avión ofrece descuentos por asientos que están vacantes.

Las spot instances proveen un descuento del 90% comparado con el precio on-demand
Las spots instances pueden ser finalizadas si la capacidad de cómputo se necesita por otros clientes on-demand
Está diseñado para aplicaciones que tiene un comienzo flexible o aplicaciones que son solamente factibles a costos de cómputo muy bajos

AWS Batch: es una manera fácil y conveniente de usar spot pricing
Condiciones de finalización: Las instancias pueden ser finalizadas por AWS en cualquier momento
Si tu instancia es terminada por AWS no se te cobra por un uso parcial de hora
Si vos finalizás una instancia se te cobra por cualquier hora en la que haya estado corriendo

-----------------------
Instancias dedicadas

Las instancias dedicadas están diseñadas para cumplir con requerimientos regulatorios

* Multi-tenant: es como cuando muchas personas viven en un mismo edificio de departamentos
Cuando múltiples clientes están corriendo cargas de trabajo en el mismo hardware. El aislamiento virtual es lo que separa a los clientes

* Single-tenant: es como cuando cada uno tiene su casa
Cuando un cliente tiene hardware dedicado. El aislamiento físico es lo que separa a los clientes

Dedicado puede ofrecerse:
* on-demand
* reservado (ahorro de hasta 60%)
* spot (ahorro de hasta 90%)

Elegís la tenencia cuando lanzás tu EC2 (dedicated host es una de las opciones)

Organizaciones grandes y empresas pueden tener preocupaciones de seguridad al compartir del mismo hardware con otros clientes de AWS

--------------------------
Planes de ahorro

Los planes de ahorro te ofrecen descuentos similares a RIs pero simplifican el proceso de compra

Hay tres tipos de planes de ahorro:
* planes de ahorro de cómputo
* planes de ahorro de instancias EC2
* planes de ahorro SageMaker

Se pueden elegir dos plazos diferentes:
* 1 año
* 3 años

Elegís las siguientes opciones de pago:
* Todo por adelantado
* Parcialmente por adelantado
* No por adelantado

Planes de ahorro de cómputo:
Proveen la mayor flexibilidad y ayudan a reducir tu costo por un 66%. Estos planes aplican automáticamente a uso de instancias EC2, AWS fargate y a uso de servicio AWS lambda sin importar la familia de instancia, tamaño, AZ, región, S.O o tenencia

Planes de ahorro de instancias EC2:
provee los precios más bajos, ofreciendo ahorro de hasta 72% a cambio de compromiso de uso de familias de instancias individuales en una región. Automáticamente reduce tu costo en la familia de instancia seleccionada en esa región sin importar AZ, tamaño, SO o tenencia. Te da flexibilidad para cambiar tu uso entre instancias dentro de una familia en esa región

Planes de ahorro de SageMaker:
Te ayuda a reducir costos de SageMaker a un 64%. Automáticamente aplica a uso de SageMaker sin importar familia de instancia, tamaño, componente o región de AWS

Identidad
----------------------------------------------------------------------------------------------
Zero trust model

Opera según el principio "no confiar en nadie, verificar todo"

Al ser los actores maliciosos capaces de sobrepasar controles de acceso convencional demuestra que las medidas de seguridad (security) tradicionales ya no son suficientes

La identidad se convierte en el perímetro principal de seguridad

¿Qué es el perímetro principal de seguridad (primary security perimeter)?
Define la primera línea de defensa y sus controles de seguridad que protegen a los recursos de nube de una compañía y sus "assets"

Network-centric (la viaje manera)
La seguridad tradicional se enfocaba en firewalls y VPNs al pocos empleados y workstations fuera de la oficina o estaban en oficinas remotas específicas

Identity-centric (la nueva manera)
Las workstations remotas son mucho más comunes, no podemos confiar en que el empleado esté en una ubicación segura, tenemos controles de seguridad basados en identidad como MFA, o provisión de acceso provisional basado en el nivel de riesgo desde donde, cuando y qué usuario quiere acceder

Identity-centric no reemplaza a network-centric, lo mejora

-----------------------------
Zero trust en AWS

Controles de seguridad de identidad que podés implementar en AWS para cumplir con el Zero trust model

AWS Identity and Access Management (IAM)
* IAM policies
* Límites de permisos (permission boundaries)
* Políticas de control de servicio (Service control policies)
* IAM policy conditions

AWS no tiene controles de identidad listos para usar que sean inteligentes, por lo cual se considera que AWS no tiene una oferta zero trust para clientes, por lo que servicios de terceros deben ser usados

AWS CloudTrail: rastrea todas las llmadas de APIs
Amazon GuardDuty: detecta actividad sospechosa o maliciosa basado en CloudTrail y otros logs
Amazon Detective: usado para analizar, investigar e identificar rápidamente problemas de seguridad (puede ingerir hallazgos de Guard Duty)

---------------------------
Zero trust en AWS con terceros

AWS técnicamente implementa un zero trust model pero no permite controles de seguridad de identidad inteligentes

Por ejemplo:
Azure active directory tiene una detección de riesgo calculada y en tiempo real basada en más puntso de datos que AWS. Está basado en: dispositivo y aplicación; hora del día; ubicación; si MFA está activado; lo que está siendo accedido.
Y los controles de seguridad, verificaciones o restricciones lógicas es mucho más robusto

Soluciones de identidad de terceros
* Azure Active Directory (Azure AD)
* Google BeyondCorp
* JumpCloud
Todos tienen más controles de seguridad inteligente para detección en tiempo real

-----------------------------
Servicio de directorio (directory service)

¿Qué es un directory service?
Un directory service mapea nombres de recursos de red a sus direcciones de red

Un directory service es infraestructura de información compartida para localizar, gestionar, administrar y organizar recursos:
* Volumes
* Carpetas
* Archivos
* Impresoras
* Usuarios
* Grupos
* Dispositivos
* Números de teléfono
* Otros objetos

Un directory service es un componente crítico de un sistema operativo de red

directory server (name server): es un servidor que provee un directory service

Cada recurso en la red es considerado un objeto por el directory server. La información sobre un recurso en particular es almacenada como una colección de atributos asociados con ese recurso u objeto

Algunos directory service bien conocidos:
* Domain name service (DNS): el directory service de internet
* Microsoft Active Directory: Azure Active Directory

-----------------------------
Active directory

Microsoft introdujo Active Directory Domain Services en Windows 2000 para darles a las organizaciones la habilidad de gestionar múltiples componentes y sistemas de infraestructura on-premise usando una identidad individual por usuario

---------------------------
Proveedores de identidad o identity providers (IpD)

Un identity provider es una entidad de sistema que crea, mantiene y gestiona información de identidad para "principals" y también provee servicios de autenticación para aplicaciones dentro de una federación o red distribuida. Un proveedor confiable de tu identidad de usuario que te deja autenticar para acceder a otros servicios. Algunos identity providers podrían ser Facebook, Amazon, Google, Twitter, GitHub, Linkedin

Federated identity es un método que vincula la identidad de un usuario a lo largo de múltiples sistemas separados de gestión de identidad

OpenID
Es un protocolo de autenticación descentralizado, por ejemplo, permite hacer login en una red social usando tu cuenta de Google o Facebook.

OAuth2.0
Es un protocolo estándar de la industria para autorización. Oauth no comparte contraseñas, en su lugar usa token de autorización para demostrar una identidad entre consumidores y proveedores de servicios. Oauth es sobre otorgar acceso a funcionalidades

SAML
Security Assertion Markup Languaje es un estándar abierto para intercambiar autenticación y autorización entre un identity provider y un service provider. Un importante caso de uso para SAML es Single-Sign-on vía navegador web

--------------------------
Single-Sign-On

SSO es un sistema de autenticación que permite al usuario ingresar con un solo ID y contraseña a diferentes sistemas y software

SSO permite a los departamentos de IT administrar una sola identidad que puede acceder a muchas máquinas y servicios en la nube

El ingreso (login) para SSO es fluido, donde una vez el usuario ha ingresado al directorio principal, mientras utilicen este software no se les presenta una pantalla de login

--------------------------
LDAP

Lightweight Directory Access Protocol es un protocolo de aplicación estándar de la industria abierto y "vendor-neutral" para acceder y mantener servicios de información de directorio distribuido en lugar de una IP de red

Un uso común de LDAP es proveer un lugar central a donde almacenar nombres de usuario y contraseñas

LDAP permite same-sign on. Esto permite a los usuarios tener un solo ID y contraseña, pero tienen que ingresarlos siempre que quieran ingresar

¿Por qué usar LDAP cuando SSO es más conveniente?
* La mayoría de los SSO están usando LDAP
* LDAP no fue diseñado para trabajar nativamente con aplicaciones web
* Algunos sistemas solo soportan integración con LDAP y no SSO

--------------------------
Multi-factor authentication

También conocido como MFA. Es un control de seguridad en el cual, luego de que llenás los campos con tu usuario/email y contraseña, tenés que usar un segundo dispositivo (como un teléfono) para confirmar que estás ingresando a tu cuenta

Te protege contra gente que pueden haberte robado la contraseña

MFA es una opción en la mayoría de los proveedores de la nube e incluso redes sociales como Facebook

----------------------------
Security keys

Es un dispositivo secundario usado como segundo paso en el proceso de autenticación para ganar acceso a un dispositivo, Workstation o aplicación

Una llave de seguridad puede recordar a un memory stick. Cuando tu dedo hace contacto con un botón o metal expuesto en el dispositivo va a generar y auto-llenar un token de seguridad

Una marca popular de llave de seguridad es Yubikey. Funciona con Gmail, Facebook y otros cientos. Soporta FIDO2/WebAuthn,  U2F. Es a prueba de agua y resistente a golpes. USB-A y NFC dual connectors en una sola llave

AWS IAM (Identity and Access management)

Podés crear y gestionar usuarios y grupos de AWS, y usar permisos para permitirles y denegarles acceso a recursos de AWS

Políticas (policies) IAM: son documentos JSON que otorgan permisos para usuarios, grupos y roles específicos para acceder a servicios. Las políticas (policies) están ligadas a identidades IAM

IAM permissions: las acciones de API que pueden o no llevarse a cabo. Están presentados en el documento de IAM policy

IAM identities:
* IAM users: usuarios finales que ingresan en la consola o interactúan con recursos de AWS programáticamente o vía click en interfaces de usuario 
* IAM groups: Se pueden agrupar usuarios para que compartan niveles de permisos del grupo, por ejemplo: administradores, desarrolladores, auditores
* IAM roles: el rol otorga permisos a recursos de AWS a acciones específicas de API de AWS. Asociar políticas a un rol y luego lo asigna a un recurso de AWS 

--------------------------
Anatomía de una IAM policy

Las IAM policies están escritas en JSON y contienen los permisos que determinan que acciones de API están permitidas o denegadas

* Version
* Statement
  - Sid (opcional): es una manera de etiquetar tus staetments
  - Effect: determina si una política va a permitir o denegar
  - Action: lista de acciones que la política permite o denega
  - Principal: cuenta, usuario, rol o usuario federado al que quisieras permitir o denegar acceso
  - Resource: el recurso al que la/s accion/es aplica/n
  - Condition (opcional): circunstancias bajo las cuales la política otorga permisos 

-------------------------
Principle of least privilige (PoLP)

PoLP es el concepto de seguridad en computadoras de proveerle a un usuario, rol o aplicación la mínima cantidad de permisos para llevar a cabo una operación o acción

Just-Enough-Acces (JEA): se le permiten solo las acciones exactas a la identidad para llevar a cabo una tarea

Just-In-Time (JIT): se le permite la más pequeña duración a una identidad para usar permisos

Risk-based adaptive policies: Cada intento de acceder a un recurso genera un puntaje de riesgo de que tan probable  es que una solicitud sea de una fuente comprometida. El puntaje de riesgo podría estar basado en muchos factores, por ejemplo: dispositivo, ubicación de usuario, la dirección IP, que servicio está siendo accedido y cuando, etc

ConsoleMe es un proyecto de código abierto de Netflix para que un usuario final pueda acceder a recursos de AWS mientras se hace cumplir JEA y JIT

Usuario root de cuenta de AWS

Cuenta de AWS: es la cuenta que tiene todos los recursos de AWS
Usuario root de cuenta de AWS: una cuenta especial con acceso total y no puede ser borrada
Usuario de cuenta de AWS: un usuario para tareas comunes que tiene permisos asignados

Tener en cuenta que cuando se habla de un cuenta de AWS, a veces se hará referencia a un usuario y otras veces a una cuenta que contiene los usuarios 

Usuario root de cuenta de AWS: es un usuario especial que se crea en el momento de la creación de una cuenta de AWS
* Usa un email y una contraseña para login, mientras que un usuario regular debe proveer el ID de la cuenta/alias, nombre de usuario y contraseña
* El usuario root no se puede borrar
* El usuario root tiene permisos totales a la cuenta y sus permisos no pueden ser limitados, no podés usar IAM policies para denegar al usuario root el acceso a recursos, pero podés usar un AWS organizations service control policy (SCP) para limitar los permisos del usuario root 
* Solo puede haber un usuario root por cada cuenta de AWS
* El usuario root es para tareas específicas que son poco frecuentes o rara vez llevadas a cabo, el usuario root no debería ser usado para tareas diarias comunes 
* Se recomienda con fuerza nunca usar las claves de acceso del usuario root
* Se recomienda con fuerza activar MFA para el usuario root

Tareas administrativas que deben ser llevadas a cabo solo por el usuario root:
* Cambiar las configuraciones de la cuenta: nombre, dirección de email, contraseña de root y claves de acceso del usuario root. Otras configuraciones de cuenta como información de contacto, divisa de pago y regiones no requieren credenciales del usuario root
* Restaurar permisos de usuario IAM. Si el único administrador IAM anula accidentalmente sus propios permisos, podés iniciar sesión como usuario root para editar las políticas y restaurar esos permisos
* Activar acceso IAM a la facturación y consola de gestión de costo
* Ver ciertas facturas de impuestos
* Cerrar tu cuenta de AWS
* Cambiar o cancelar tu plan de soporte de AWS
* Registrarte como vendedor en el mercado de instancia reservada
* Permitir borrado MFA en un bucket de S3
* Editar o borrar una política de amazon S3 bucket que incluya un VPC ID o VPC endpoint ID inválido
* Inscribirse a GovCloud

Integración de aplicaciones

¿Qué es?
Es el proceso de permitir que dos aplicaciones independientes se comuniquen y trabajen entre sí, comúnmente facilitado por un sistema intermediario

Las cargas de trabajo de la nube alientan a los sistemas y servicios a estar débilmente acoplados y así  AWS tiene muchos servicios para el propósito específico de integración de aplicaciones

Los sistemas comunes o patrones de diseño utilizados para integración de aplicaciones generalmente son:
* Queueing
* Streaming
* Pub/Sub
* API gateways
* State machine
* Event bus  	

----------------------------
Queueing

¿Qué es un sistema de mensajería?
Se usa para proveer comunicación asincrónica y desacoplar procesos vía mensajes/eventos de un emisor y receptor (productor y consumidor)

¿Qué es un sistema de colas (queueing system)?
Un sistema de colas es un sistema de mensajería que generalmente borrará mensajes una vez se consuman. Comunicación simple. No es en tiempo real. Hay que "tironear" (pull) los datos. No reactivo

Simple Queueing Service (SQS)
Un servicio de colas completamente gestionado que te permite desacoplar y escalar microservicios, sistemas distribuidos y aplicaciones serverless

Caso de uso: necesitás hacer cola (queue up) de emails de transacción para ser enviados, por ejemplo: registrarse, reinicializar contraseña

-------------------------------
Streaming

¿Qué es?
Múltiples consumidores pueden reaccionar a eventos (mensaje)
Los eventos viven en el stream por largos períodos de tiempo, así operaciones complejas pueden ser aplicadas. Es en tiempo real

Amazon Kinesis
Es una solución completamente gestionada para recolectar, procesar y analizar streaming de datos en la nube
