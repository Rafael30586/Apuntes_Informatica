Introducción a Apache Tomcat

Apache Tomcat es un servidor web open-source y servlet container desarrollado por Apache Software Foundation. Aunque generalmente se lo llama "servidor de aplicación", técnicamente es un servidor web diseñado específicamente para ejecutar servlets java y reproducir (renderizar) JSP.
A diferencia de un servidor Java EE de aplicación (como JBoss o WebLogic), que soporta el stack entero de funciones empresariales (EJB, JMS, etc), Tomcat es liviano y enfocado puramente en la parte web. Esto hace que sea el estándar de la industria para alojar aplicaciones desarrolladas con spring boot y microservicios.

------------------
Componentes clave

1. Catalina (el servlet container)
Catalina es el motor que implementa la especificación servlet java. Gestiona el ciclo de vida de tus servlets (init, service, destroy). Cuando desplegás un archivo .war, Catalina lo desempaqueta y carga las clases.
2. Coyote (el conector)
Coyote se encarga de la comunicación de red. Soporta protocolos HTTP/1.1, HTTP/2 y AJP. Gestiona el thread pool, creando hilos para manejar eficientemente las solicitudes entrantes.
3. Jasper (el motor JSP)
Si tu aplicación usa archivos .jsp, Jasper es responsable de "parsearlas". Compila el archivo JSP a un servlet java estándar (archivo .class) para que Catalina pueda ejecutarlo. Por esto es que la primera carga de una página JSP es lenta (compilación) pero las cargas siguientes son rápidas. 

-------------------
Arquitectura central: como Tomcat es construido

Tomcat es construido de manera jerárquica. Entender esta jerarquía es clave para configurarlo correctamente.
1. Servidor: El componente más importante. Representa la instancia de Tomcat entera (JVM). Está a la escucha esperando por un comando de cerrado en un puerto específico (por defecto el 8005)
2. Servicio: Un grupo de uno o más conectores que comparten un solo motor. Rol: hace de puente entre el afuera (conectores) con la lógica de procesamiento interna (motor)
3. Conector (Coyote): El componente que está a la escucha de conexiones entrantes
* Conector HTTP: escucha en el puerto 8080 (por defecto) para tráfico web estándar 
* Conector AJP: escucha en puerto 8009 por tráfico desde un proxy reverso como Apache HTTPD
4. Motor (Catalina): el cerebro de la operación. Representa la tubería (pipeline) de procesamiento de solicitudes. Examina las cabeceras de la solicitud entrante para determinar que anfitrión y contexto debería encargarse de él.
5. Anfitrión: representa un anfitrión virtual (por ejemplo: www.ejemplo.com). Tenés múltiples anfitriones en un servidor tomcat.
6. Contexto: representa una aplicación web (por ejemplo: /my-app)

-------------------
El ciclo de vida de la solicitud: del navegador al servlet

¿Qué ocurre cuando un usuario escribe http://localhost:8080/my-app/hello ?
1. Acepta: el conector coyote a la escucha del puerto 8080 acepta la conexión TCP
2. Parse: "parsea" los encabezados HTTP y crea objetos HttpServletRequest y HttpServletResponse
3. Enruta (motor): el conector pasa estos objetos al motor, que mira al nombre de anfitrión (localhost)
4. Enruta (anfitrión): el motor pasa la solicitud al anfitrión a tono (matching host)
5. Enruta (contexto): el anfitrión mira a la ruta de la URL (/my-app) y pasa la solicitud al contexto correcto (tu aplicación)
6. Ejecuta (servlet): el contexto mira al mapeo específico (/hello) e invoca al correspondiente código de java servlet
7. Respuesta: el servlet escribe datos al objeto de respuesta, que viaja de vuelta a través de la cadena (contexto -> anfitrión -> motor -> conector ) y enviado de vuelta al navegador de usuario

-------------------
Estructura de directorios de tomcat

Saber en donde los archivos están es crítico para la asministración
* /bin: scripts de puesta en marcha (startup.sh) y paro (shutdown.sh)
* /conf: archivos de configuración
* server.xml: la configuración principal (puertos, conectores, anfitriones)
* web.xml: configuraciones por defecto para todas las aplicaciones (tipos MIME, fin de sesión)
* tomcat-users.xml: usuarios y roles para el GUI gestor de tomcat
* /lib: archivos jar globales compartidos por todas las aplicaciones (por ejemplo, drivers de bases de datos)
* /logs: registros del servidor (catalina.out, localhost_access_log)
* /webapps: acá es donde desplegás tus aplicaciones. Dejá un archivo .war acá y tomcat va a desplegarlo automáticamente
* /work: directorio temporal en donde Jasper almacena servlets JSP compilados

--------------------
Casos de uso de Apache Tomcat

* Aplicaciones de spring boot: Aunque spring boot tiene un tomcat incorporado, por debajo, está corriendo el mismo motor tomcat descrito más arriba
* Microservicios: porque es liviano y comienza rápido, es ideal para microservicios en contenedores (docker)
* Aplicaciones empresariales legacy: alojando aplicaciones java tradicionales .war que usan servlets y JSPs
