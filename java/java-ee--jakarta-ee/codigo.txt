Cuando creamos un servlet, este es una clase de java que hereda de HttpServlet 

En un servlet podemos crear un método que tenga como parámetros un HttpServletRequest y HttpServletResponse. Estas son la solicitud y la respuesta que se le hacen al servlet respectivamente. 

Del objeto HttpServletRequest podemos obtener valores que provienen de una página web, como un formular, con el método getParameter(). El valor obtenido es un String 

Del objeto HttpServletResponse podemos obtener un objeto de clase PrintWriter gracias al método getWriter(). Este objeto permite que el servlet imprima en pantalla del navegador lo que nosotros queramos gracias al método println()

Los servlets se pueden mapear a un endpoint gracias a la etiqueta servlet-mapping que podemos usar en el archivo web.xml. También podemos colocarle un nombre al servlet gracias a la etiqueta servlet

En un servlet podemos usar los métodos doGet y doPost, que son para solicitudes get y post respectivamente. Amobs tienen los parámetros HttpServletRequest y HttpServletResponse. También tenemos el método service que acepta los mismos parámetros y también se puede usar para solicitudes

Podemos hacer que un servlet llame a otro servlet con el método getRequestDispatcher() (acepta como argumento el nombre del servlet al que queremos llamar) de la solicitud que devuelve un objeto RequestDispatcher. Luego con este objeto podemos usar el método forward() que acepta la solicitud y la respuesta como argumento

Podemos pasar cookies de un servlet a otro gracias a la clase Cookie del paquete jakarta.servlet.http . Las cookies se asignan a la respuesta del servlet con el método addCookie(). Luego podemos obtener las cookies de una respuesta gracias al método getCookies() que nos devuelve un array de Cookies

Con la etiqueta context-param en el archivo web.xml podemos establecer parámetros que son compartidos por todos los servlets

Con las clases ServletConfig y ServletContext podemos trabajar con parámetros definidos en el archivo web.xml. Con servletContext trabajamos con parámetros compartidos por todos los servlets y con ServletConfig trabajamos con parámetros específicos de cada servlet. Estos parámetros específicos se establecen con la etiqueta init-param dentro de la etiqueta servlet. Para obtener una instancia de ServerContext y ServerConfig se utilizan los métodos getServletContext y getServerConfig respectivamente, que son heredados por la clase HttpServlet de la clase GenericServlet

No es necesario utilizar el archivo web.xml para hacer mapeos entre servlets y endpoints, en su lugar se pueden usar anotaciones, por ejemplo, la anotación @WebServlet("/<endpoint>") en el servlet sirve para asignar un endpoint al servlet

Cuando trabajamos con jsp podemos escribir código java en los mismos encerrando dicho código entre <%  %> . Lo que está ancerrado entre esos símbolos se llama scriptlet y sería en un servlet lo que está en el método service. Si queremos que parte de nuestro código en el JSP esté fuera del método service, usamos <%! %> (se llama declaration). Para importar paquetes usamos la etiqueta <%@ import=<paquete> %> (se llama directive). Si se quiere importar varios paquetes o clases, se separa los impórts por una coma. También se puede usar la etiqueta expresion para guardar lo que quweramos en el método out.print(), se usa <%= <variable> %> 

Con jsp viene por defecto un objeto llamado request, del cual podemos obtener los argumentos con el método getParameter(), y el objeto out, que nos permite usar el método print() para imprimir valores de variables

Podemos crear filtros y mapearlos a los servlets en el archivo web.xml. En jakarta, los filtros heredan de HttpFilter e implementan Filter. Podemos hacer un casting del ServletRequest que reciben como argumentos a HttpServletRequest para procesar lo que recibe de la solicitud.

Para crear un login usando jakarta podemos usar session o cookies. Las cookies se guardan en el navegador y la session en el servidor.

Cuando colocamos una etiqueta "a" en un JSP para que sirva de enlace, podemos colocarle en href el nombre del JSP al que nos lleva, por ejemplo <a href=videos.jsp></a>

El protocolo HTTP tiene caché, lo que puede resultar en que el usuario entre a páginas que necesiten autenticación sin estar autorizado, por ejemplo, hacer logout y luego apretar el botón de ir para atrás en el navegador y que nos lleve a una página para la cual ya no tenemos autorización por no estar autenticados. Para evitar esto se puede colocar esta pieza de código en algunos JSPs, response.setHeader("Cache-Control", "no-cache, no-store, must-revalidate"); Nota: esto sirve con la versión 1.1 de HTTP, si estamos trabajando en la versión 1.0 de HTTP, hay que agregar además, la siguiente línea de código: response.setHeader("Pragma", "no-cache");

Estos símbolos <% %> nos sirven para escribir código java en el JSP y se llaman scripplets

Cuando usamos request.getAttribute() en nuestro JSP, tendremos que hacer una refundición, ya que el método devuelve un objeto de clase Object

El método init() de los filtros se llama solo una vez en todo el ciclo de vida del filtro

A las respuestas HTTP que producen los servlets se les puede asignar el "content type" con el método setContentType que implementan de la interfaz ServletResponse

A los filtros que creamos debemos asignarles una url o un servlet, no hacen falta los dos. Si queremos que un filtro se ejecute antes que determinado servlet, mapeamos al filtro con la url del servlet

La solicitud y la respuesta atraviesan el filtro, llegan al servlet y después vuelven a pasar por el filtro. Lo que escribamos después del método doFilter se ejecutará después de que la solicitud y la respuesta pasen por el servlet y vuelvan al filtro 

El método init de los filtros se ejecuta una vez cuando ponemos la aplicación en ejecución. El método destroy se ejecuta cuando detenemos la aplicación

Cuando utilizamos la etiqueta filter-mapping en el archivo web.xml, podemos utilizar un atributo llamado dispatcher que tiene valores enumerados (el enum en la API de jakarta aparece como DispatcherType). Esos valores enumerados son: RESQUEST, solo cuando una solicitud viene directamente del cliente; ASYNC, solo cuando la solicitud asíncrona viene del cliente; FORWARD, solo cuando la solicitud ha sido reenviada a un componente; INCLUDE, solo cuando la solicitud ha sido procesada por un componente que ha sido incluido; ERROR, solo cuando la solicitud está siendo procesada con el mecanismo de error de página. El DispatcherType sirve para limitar el alcance del filtro

Un solo filtro puede mapearse a más de un servlet si así lo deseamos. Si hacemos esto, se producirá una cadena de filtros. El objeto de la solicitud (ServletRequest) atravesará los filtros en un orden que dependerá de como los hayamos escrito en el archivo web.xml. Inlfuirá el orden en el cual hayamos colocado a los filtros pero también el tipo de mapeo que hayamos realizado: el mapeo a una URL tiene prioridad sobre el mapeo a un servlet

Utilizando la anotación @WebServlet se puede mapear un servlet a más de una UR de la siguiente manera: @WebServlet({"/url1","/url2"})

Podemos mapear un filtro a todas las urls de la aplicación si colocamos "/*" en el url-pattern del mapeo del filtro en el archivo web.xml

Para configurar filtros en la aplicación, se puede usar la anotación @WebFilter, que tiene muchos atributos: filterName (string), value (String[]), urlPatterns (String[]), dispatcherTypes (dispatcherType[]), servletNames (String[]), displayName (String), description (String), asyncSupported (boolean), initParams (WebInitParam[]), smallIcon (String), largeIcon (String). El atributo value o urlPatterns es obligatorio, el resto son opcionales

Cuando utilizamos la anotación @WebFilter, no podemos establecer el orden de los filtros usando anotaciones. El orden de los filtros es determinado por el nombre de la clase en orden alfabético y NO por lo que coloquemos en filterName

Cuando usamos el método sendRedirect de las solicitudes al servlet, conviene colocar la ruta sin barras, sobre todo, sin la barra inicial, por ejemplo, colocar request.sendRedirect("login") en lugar de request.sendRedirect("/login")

Un método bastante útil de la interfaz HttpServletRequest es getContextPath, que sirve para obtener la ruta del contexto de la aplicación
