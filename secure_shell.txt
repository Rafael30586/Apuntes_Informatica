¿Qué es Secure Shell (SSH) y como funciona?

Secure Shell es un protocolo de red que les da a los usuarios, particularmente a administradores de sistemas, una manera segura de acceder a una computadora en una red insegura
SSH también se refiere a un conjunto de utilidades que implementan el protocolo SSH. Secure Shell provee fuerte autenticación con contraseña y autenticación con clave pública, así como comunicaciones de datos encriptados entre dos computadoras conectándose en una red, tal como internet. SSH es importante para mantener la seguridad de los sistemas, ya que el protocolo funciona como una manera segura para proveer acceso y gestión de sistemas en red
Además de proveer fuerte encriptación, SSH es ampliamente usado por administradores de red para gestionar sistemas y aplicaciones de manera remota, permitiéndoles iniciar sesión en otra computadora de una red, ejecutar comandos y mover archivos de una computadora a otra
SSH se refiere tanto al protocolo de red criptográfico como al conjunto de utilidades que implementan el protocolo. SSH usa el modelo cliente-servidor conectando una aplicación cliente Secure Shell, que es el fin en donde la sesión es mostrada, con un servidor SSH, que es el fin en donde la sesión corre. Las implementaciones de SSH generalmente incluyen soporte para protocolos de aplicación usados para emulación de terminal o transferencia de archivos
SSH puede también ser usado para crear túneles seguros para otros protocolos de aplicación, por ejemplo, para correr de manera segura X sesión gráfica de un sistema Windows de manera remota. Un servidor SSH, por defecto, escucha en el puerto estándar 22 TCP

------------------------
¿Cómo funciona SSH?

Secure Shell fue creado para reemplazar la emulación de terminal insegura o programas de inicio de sesión, como telnet, login remoto (rlogin) y shell remoto (rsh). SSH permite las mismas funciones: correr sesiones de terminal en sistemas remotos. SSH también reemplaza programas de transferencia de archivos, como FTP y remote copy (rcp)
SSH corre en el conjunto de protocolos TCP/IP en una arquitectura cliente-servidor. SSH provee encriptación y autenticación usando una criptografía de clave pública. El uso más básico de SSH es conectarse a un anfitrión remoto para una sesión de terminal. La forma de ese comando, como ejemplo, se ve así:

ssh UserName@SSHserver.example.com

Este comando hace que el cliente intente conectarse al servidor llamado server.example.com, usando el ID de usuario UserName. Si esta es la primera vez en negociar una conexión entre en anfitrión local y el servidor, el usuario es estimulado como sigue con la huella dactilar de la clave pública del anfitrión remoto a conectar, aún sin haber tenido conexión previa:

The authenticity of host 'sample.ssh.com' cannot be established.
DSA key fingerprint is 01:23:45:67:89:ab:cd:ef:ff:fe:dc:ba:98:76:54:32:10.
Are you sure you want to continue connecting (yes/no)?

Contestar sí al aviso causa que la sesión continúe, y la clave del anfitrión es almacenada en el archivo "known_hosts" del sistema local. Este es un archivo escondido almacenado por defecto en un directorio escondido llamado /.ssh/known_hosts , en el directorio home del usuario. Una vez que la llave del anfitrión haya sido almacenada en el archivo known_hosts, el sistema cliente puede conectarse directamente a ese servidor otra vez sin la necesidad de aprobación; la clave del anfitrión autentica la conexión

------------------------
¿Para qué se usa SSH?

Presente en todos los datacenters, SSH embarca por defecto con todo servidor con UNIX, Linux o Mac. Las conexiones SSH se usan para asegurar muchos tipos diferentes de comunicaciones entre máquinas locales y anfitriones remotos, incluyendo acceso remoto seguro a recursos, ejecución de comandos remota, envío de parches de software, actualizaciones y otras tareas administrativas o de gestión 
Además de crear un canal seguro entre computadoras locales y remotas, SSH se usa para gestionar routers, hardware de servidor, plataformas de virtualización y sistemas operativos, así como gestión de sistemas internos y aplicaciones de transferencia de archivos
Secure Shell se usa para conectar servidores, hacer cambios, llevar a cabo subidas y salir, ya sea usando herramientas o directamente a través de la terminal. Claves SSH pueden ser usadas para automatizar acceso a servidores y generalmente se usan en scripts, sistemas de respaldo y herramientas de gestión de configuración
Diseñadas para ser convenientes y funcionar a lo largo de fronteras organizacionales, las claves SSH proveen "single sign-on" (SSO) para que los usuarios puedan moverse entre sus cuentas sin tener que escribir la contraseña cada vez
SSH puede usarse también para crear túneles seguros para otros protocolos de aplicación. Por ejemplo, túneles seguros pueden para envío a puertos local, remoto y dinámico. Mientras se juega en roles giratorios en gestión de identidad y acceso, SSH hace más que autenticar en una conexión encriptada. Todo el tráfico de SSH está encriptado. Ya sea que los usuarios transfieran un archivo, naveguen por la web o ejecuten un comando, sus acciones son privadas	 
Aunque es posible usar SSH con un ID de usuario ordinario y contraseña, SSH depende más seguido de una autenticación basada en clave usando pares de claves públicas para autenticar anfitriones entre sí. Usuarios individuales deberán emplear su ID de usuario y contraseña, u otros métodos de autenticación de usuario, para conectarse al anfitrión remoto, pero la máquina local y la máquina remota se autentican separadas la una de la otra. Esto se logra al generar un único par clave pública para cada anfitrión en la comunicación. Una sola sesión requiere 2 pares de clave pública: un par clave pública para autenticar la máquina remota en la máquina local y un segundo para de clave pública para autenticar la máquina local a la máquina remota
SSH también es usado para gestionar y configurar firewalls u otras aplicaciones de seguridad, asegurando cambios y acceso de configuración administrativa

Capacidades de la secure Shell

SSH permite las siguientes funciones:
* Acceso remoto seguro a sistemas de red con SSH activado o dispositivos para usuarios, así como procesos automatizados
* Sesiones de transferencia de archivos segura e interactiva
* Transferencias de archivos seguras y automatizadas
* Emisión segura de comandos en dispositivos o sistemas remotos
* Gestión segura de componentes de infraestructura de red
* Autenticación basada en anfitrión
* Ejecución remota de comandos
* Envío por puertos (Port forwarding)
* Gestión de clave y sesión

SSH puede ser usado de manera interactiva para permitir sesiones de terminal y debería ser usado en lugar del menos seguro programa Telnet. SSH es también comúnmente usado en scripts y otro software para permitir a programas y sistemas acceder de manera segura y remota a datos y otros recursos

---------------------------
Problemas de seguridad de SSH

Empresas usando SSH deberían considerar encontrar formas de gestionar claves de anfitriones almacenados en sistemas clientes. Estas claves se pueden acumular con el tiempo, especialmente para staff IT que necesita acceder a anfitriones remotos para propósitos de gestión
Ya que los datos almacenados en un archivo known_hosts pueden ser usados para obtener acceso autenticado a sistemas remotos, las organizaciones deberían estar enteradas de su existencia y tener un proceso estándar para retener control sobre ellos, incluso cuando el sistema es sacado de comisión, y los discos duros podrían tener estos datos almacenados en texto plano
Los desarrolladores deberían ser cuidadosos al incorporar comandos SSH o funciones en un script u otro tipo de programa. Aunque es posible emitir un comando SSH que incluya ID de usuario y contraseña para autenticar al usuario de la máquina local a una cuenta en el anfitrión remoto, hacerlo puede exponer credenciales a un atacante que tenga acceso al código fuente
Un hoyo de seguridad en el procesador de comandos Bash, puede ser ejecutado sobre SSH, pero su vulnerabilidad está en Bash, no en SSH
La mayor amenaza a SSH es una pobre gestión de claves. Sin la apropiada creación, rotación y quitado centralizado de claves SSH, las organizaciones pueden perder control sobre quien tiene acceso a que recurso y cuando, particularmente cuando SSH se usa en procesos automatizados aplicación-a-aplicación

SSH vs Telnet

Telnet fue uno de los primeros protocolos de aplicación de internet. Se usa para iniciar y mantener una emulación de terminal en un anfitrión remoto
SSH y Telnet son similares, con la diferencia principal siendo que SSH usa criptografía de clave pública para autenticar endpoints al configurar una sesión de terminal, así como encriptar comandos y salida de sesión
SSH encripta todos los datos transmitidos entre el cliente y el servidor, lo que incluye contraseñas y comandos. Telnet transmite los datos en texto plano, haciéndolo muchos más vulnerable. SSH también soporta múltiples métodos de autenticación, incluyendo clave pública, contraseña y autenticación multifactor, mientras que Telnet usa típicamente autenticación por usuario y contraseña. Mientras telnet es usado principalmente para emulación de terminal, SSH puede ser usado para emulación de terminal (similar al comando rlogin) así como para emitir comandos de manera remota como rsh, transfiriendo archivos usando SSH FTP (SFTP) y tunelizando otras aplicaciones 

----------------------------
SSH vs SSL/TSL

El protocolo de Transport Layer Security (TLS), que actualiza al protocolo Secure Sockets Layer (SSL), fue diseñado para proveer seguridad para transmisiones de red en la capa de transporte. El protocolo SSH también opera en o un poco más arriba de la capa de transporte, pero hay diferencias importantes entre los dos protocolos.
Aunque ambos dependen de pares de claves públicas o privadas, solo el servidor es autenticado con un par de claves bajo TLS. SSH usa un par de claves separado para autenticar cada conexión: un par de claves para una conexión de una máquina local a una máquina remota y un segundo par de claves para autenticar la conexión de una máquina remota a una máquina local.
Otra diferencia entre SSH y TLS es que TLS puede estar encriptado sin requerir autenticación y puede autenticar sin encriptación. SSH autentica y encripta todas las conexiones
SSH provee a los profesionales de IT e información con un mecanismo seguro para gestionar clientes SSH de manera remota. En lugar de requerir autenticación con contraseña para inicializar una conexión entre un cliente SSH y un servidor, SSH autentica a los propios dispositivos. Esto le permite al staff de IT conectarse con sistemas remotos y modificar configuraciones de SSH, incluyendo añadir o quitar pares de claves de anfitriones en el archivo known_hosts

Implementaciones de SSH

SSH es un protocolo abierto. Ha sido implementado para la mayoría de las plataformas de computación. La implementación de OpenSSH de código abierto se encuentra en general en Linux, UNIX y otros sistemas operativos basados en BSD, incluyendo a MacOS de Apple
OpenSSH fue porteado para correr en Windows powershell comenzando en 2015. En 2018, soporte opcional para OpenSSH fue añadido a windows 10. Mientras SSH es accesible directamente por defecto en la mayoría de los sistemas operativos tipo Unix, la versión porteada de Windows de OpenSSH debe ser explícitamente habilitada en las configuraciones de Windows
PuTTY es otra implementación de código abierto de SSH. Aunque está actualmente disponible para Windows, macOS, Unix y BSD, PuTTY fue originalmente escrito para correr en Windows. Ha sido por mucho tiempo una de las mejores opciones para usar SSH en un sistema windows 
La mayoría de las implementaciones del paquete SSK abarcan las siguientes tres utilidades:
1. Inicio de sesión seguro (slogin)
2. ssh
3. copia segura (scp)
Estas son versiones seguras de las anteriores utilidades inseguras de Unix rlogin, rsh y rcp
SSH usa criptografía de clave pública para autenticar la computadora remota y habilitar la computadora remota para autenticar el usuario si es necesario
Actualmente hay decenas de implementaciones de SSH disponibles para varias plataformas y bajo una variedad de licencias propietarias y de código abierto
